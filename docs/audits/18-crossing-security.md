## Findings

### [medium] apps/worker/src/composition/subscription-auth-guard.ts:43 — shadowing-env guard omits lowercase proxy vars + `ANTHROPIC_CUSTOM_HEADERS` (automated-backstop fail-OPEN)
`SUBSCRIPTION_SHADOWING_ENV_KEYS` watches the 8 uppercase vars but NOT `http_proxy`/`https_proxy` (lowercase — respected by `proxy-from-env`-style resolution used by many Node HTTP stacks / the spawned Claude Code CLI), `ALL_PROXY`/`all_proxy`, nor `ANTHROPIC_CUSTOM_HEADERS` (documented Claude Code header-injection vector). On the ARMED path a mis-set lowercase proxy would NOT refuse the arm, so permitted subscription egress could route through an unexpected proxy (egress-integrity/MITM for content the veto already permitted).
- **Why not crit/high / not BLOCKED:** this does NOT create an employer-raw leak (the egress veto independently DENIES employer-raw+ack-OFF to any cloud route BEFORE the run leg, proven at the assembled broker), does NOT uncap spend (`maxBudgetUsd` is independent), and does NOT admit an unvalidated/inferred write (the broker gate + validateNoInference are independent). It weakens only the defense-in-depth backstop for ALREADY-PERMITTED egress.
- **Mitigations already in place:** the code comment names the set as EXTENSIBLE and "re-verify against the LIVE Agent-SDK docs at the flip"; the runbook CHECKPOINT-1 makes the out-of-band environment verification a HARD, owner-owned flip precondition; the maiden run was benign non-employer content.
- **Action:** `step-9-flag` (Finding, medium) → extend the watched set (`http_proxy`, `https_proxy`, `ALL_PROXY`, `all_proxy`, `NO_PROXY`, `ANTHROPIC_CUSTOM_HEADERS`) BEFORE any employer-raw armed run; retroactively confirm (out-of-band) that no lowercase-proxy/custom-header var was set for the maiden run.
