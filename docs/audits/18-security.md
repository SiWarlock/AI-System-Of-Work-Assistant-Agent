## Findings

### MEDIUM-1 — `buildWorkspace` posture not re-scoped to the source's routing-bound target workspace (latent rule-5/WS-8 hole under GATE-2 multi-ws arming)
`apps/worker/src/composition/buildActivities.ts` (source `sourceAgent` wiring, ~L735–790) scopes `buildMatrix`/`buildEgress` to `ctx.workspaceId ?? params.resolved…`, but **`buildWorkspace: () => ({ type: params.resolved.type, dataOwner: params.resolved.dataOwner })` ignores `ctx`** — it always returns the WORKER-DEFAULT workspace posture. The broker's `vetoJobEgress(job, route, egress, workspace, …)` keys the rule-5 employer-raw veto off `workspace.type`. In a **multi-workspace-per-worker** deployment where a source classifies (via the real registry resolver) into a workspace of a DIFFERENT type than the worker default, the veto would evaluate an EMPLOYER target under a non-employer posture ⇒ rule-5 could be skipped for that path.

- **Reachability today: INERT.** Requires the full stack of owner-gated conditions to coincide: (1) GATE-2 multi-workspace arming, (2) the real registry content resolver bound (dormant — shipped default is `createBootWorkspaceContentResolver`, single boot ws, `ctx.workspaceId === params.resolved.workspaceId`), (3) `providerTransport` armed with a real CLOUD route for `source.process` (owner crossing; shipped `source.process` route is loopback-local ollama), (4) an employer target with ack OFF. Under the shipped single-workspace-per-worker topology `buildWorkspace` is correct by construction (one workspace). The in-code comment already flags the matrix/egress scoping as "a NO-OP … only bites when a source classifies into a workspace distinct from the worker's default."
- **Severity: medium.** Not a phase-exit blocker (dormant; the phase is safe-build). It is a real latent bypass surface the per-slice reviews of 18.4 (source routing) and 18.9 (egress veto) could not connect because they are separate slices.
- **Action:** `step-9-flag` → **GATE-2 crossing checklist (Finding).** When multi-workspace arming is done, `buildWorkspace` MUST derive `{type, dataOwner}` from the source's routing-bound TARGET workspace (registry-sourced, same axis as the matrix/egress re-scope), so the rule-5 veto evaluates the correct posture. Add a multi-ws assembled test: an employer-typed target source + ack-OFF cloud route ⇒ DENY at `egress_veto`.

### LOW-1 — §8 dispatch-time `payloadHash` re-hash must reconstruct the identical preimage (forward verification, Phase 21/22)
`apps/worker/src/composition/proposed-action-producer.ts:105` stamps `payloadHash = "payload:" + targetSystem + ":" + sha256(JSON.stringify(payload))`. The §8 TOCTOU guard (dormant — dispatch is Phase 21/22) must re-derive the hash with the IDENTICAL preimage form to compare against the frozen `Approval.payloadHash`; a divergent re-hash fails CLOSED (blocks dispatch — safe direction) but silently breaks every dispatch. Also `payload` is a fixed 2-key literal (`{operation, intent}`) so `JSON.stringify` is deterministic today; a future dynamic-key payload could reorder keys and break the compare. **Action:** `defer` — record on the dispatch crossing checklist (pin the producer↔§8-guard hash-preimage contract with a round-trip test at that wave).

### LOW-2 — `produceProposedActions` relies on the `ValidateExtractionPort` structural invariant for `validated.fields` (defensiveness asymmetry)
`proposed-action-producer.ts:78` reads `validated.fields[binding.actionIntentField]` directly, whereas the sibling 18.8 frontmatter code defensively degrades `validated.fields ?? {}`. If `validated.fields` were contract-violatingly `undefined` AND a binding were configured (dormant), the producer would throw — but this cannot occur in practice: the upstream `createMeetingExtractionSchemaGate` rejects a null/non-object/empty `fields` before `validated` is produced, and any throw fails CLOSED (no note, no proposal, no unsafe side effect). **Action:** `defer` — optional consistency nit; mirror the `?? {}` guard for symmetry if touched.
