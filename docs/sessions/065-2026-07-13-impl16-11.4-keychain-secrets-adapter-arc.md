## What was built — 3 slices

### Slice 1 — adapter core (`4010155`, brief 049)
`apps/worker/src/secrets/keychain-adapter.ts` (NEW) — `createKeychainSecretsAdapter(backend): SecretsPort` over a mockable `KeychainBackend` seam. Parses `keychain://<service>/<account>` fail-closed (charset + traversal + length guards, no backend call on malformed → `invalid_ref`); maps every backend fault to a typed `SecretUnresolved` carrying only `{code, ref, reason}` (a fixed `KeychainUnresolvedReason` token — never key bytes or raw stderr); rejects a zero-length key; `Object.hasOwn`-guards the reason map (no prototype-chain read); never throws. Tests over a fake in-memory backend.

### Slice 2 — security-CLI backend (`4db092e`, brief 050)
`apps/worker/src/secrets/keychain-backend.ts` (NEW) — `createSecurityCliKeychainBackend({exec})` runs `security find-generic-password -w -s <svc> -a <acct>` over an injected execFile-shaped exec (args-array, no shell, absolute `/usr/bin/security`). exit 0 → `ok(Uint8Array)` = the `-w` stdout minus one trailing `\n`, **de-aliased via `new Uint8Array(subarray)`** (not `.slice` — a Node Buffer's slice shares memory); fault → typed kind (44→not_found, stderr-pattern locked/denied, else backend_error); `detail` bounded + secret-scrubbed; the fault path reads only code+stderr (stdout can't reach `detail` by construction); never throws. Tests over a fake exec.

### Slice 3 — boot wiring capstone (`536d3b2`, brief 051)
`apps/worker/src/secrets/keychain-boot.ts` (NEW) — `buildKeychainSecrets(gate)` constructs the adapter+backend ONLY when the new `BootConfig.keychainSecrets` gate is present (else `undefined` = inert), exposing the `SecretsPort` + a `getSecret` `SecretsAccessor` facade. `apps/worker/src/boot.ts` (MODIFIED) sources `provenanceServingOracle.secrets = keychainSecrets?.secrets ?? provenanceBundle.secrets` (Keychain wins; inline is a test fallback; `.secrets` made optional). The real bounded no-shell execFile wrapper (`createRealExecFile`) is built only on the provisioned path; `mapExecResult` (exported, unit-pinned) maps the callback triple with the load-bearing "a fault never maps to exit 0". Default (gate absent) is byte-equivalent — no `security` process.

## Decisions made

- **Slice 1 / Q3 — a 5th `invalid_ref` reason token** (distinct from the 4 backend-fault tokens): a config error, not a Keychain runtime fault (the degraded controller distinguishes non-retryable misconfig from retryable-on-unlock). Orch-endorsed.
- **Slice 2 / `--` OMITTED** — a corrective on orch's Slice-1 carry-forward: `find-generic-password` takes no positional we pass; `-s`/`-a` values are consumed literally by getopt regardless of a leading `-`, so argv-injection is already structurally impossible (args-array + absolute bin + Slice-1 charset guard). Orch confirmed.
- **Slice 2 / de-alias via `new Uint8Array(subarray)`** — a review-caught MEDIUM: a Node `Buffer` (what the real execFile yields) overrides `Uint8Array.slice` with view semantics, so `.slice` would re-alias the pool on the real-secret path. Fixed + Buffer-input test.
- **Slice 3 / Q3 — Keychain-locked degraded routing DEFERRED to Slice 4** — the `KeychainLockController` is not boot-wired and the resolution call-sites are dormant, so folding the routing in would wire a controller + a live call-site into dormant paths with nothing to exercise (sequencing, not a scope cut). Orch confirmed.
- **Slice 3 / `provenanceServingOracle.secrets` made optional** + Keychain-wins sourcing — a small backward-compatible change to my own C5.4b worker-internal BootConfig type.
