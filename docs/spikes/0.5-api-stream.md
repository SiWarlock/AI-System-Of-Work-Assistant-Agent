# Spike 0.5 — Local API streaming primitive (status: GO-WITH-CONDITIONS)

> OQ-002 · Spec anchors: `ARCHITECTURE.md` §10 (Local App API — the worker→renderer push stream), §5 (renderer↔worker per-launch session-token auth + Origin/Host allowlist), §16 (worker supervision/restart + System Health), §1/§4 (loopback-only transport). Plan task 0.5 + downstream 11/“single push stream (WS or SSE, per the Phase-0 API spike)”.
> Date run: 2026-06-29 · Host: macOS 25.4.0, arm64 (Apple Silicon) · Node v22.13.0 · pnpm 11.5.2.

## Question / unknown

What is the worker→renderer **push primitive** for the §10 stream (workflow status, approval updates, System Health, read-model changes): a **tRPC v11 subscription over WebSocket** (`wsLink` + `ws`) or an **SSE-style stream** (`httpSubscriptionLink`)? And under the §5 **per-launch session-token** auth model (token minted by Electron main, injected only via preload, required on the *stream handshake*, with an Origin/Host allowlist), do both survive **forced disconnect → reconnect** with **no silently-dropped events** (the spec’s “reconnection/backpressure semantics are pinned in the API spike”)?

**NO-GO branch named by the plan:** the two are interchangeable for §10 — the loser is the validated fallback (see No-go branch).

## What I did

Throwaway spike in `…/scratchpad/phase0/0.5/` (NOT in the repo). Real toolchain, real packages, real sockets:

1. **One adapter-agnostic tRPC v11 router** (`shared.mjs`) modelling the §10 stream: a single `onEvent` subscription that yields a **monotonic seq** wrapped in **`tracked(id, data)`**, backed by a **server-side replay log** so a reconnect resumes from `input.lastEventId` losslessly. The **same router** is mounted under **both** adapters — so the transport choice is the *only* variable.
2. **§5 auth in `createContext`, pre-handler:** the per-launch session token rides **`connectionParams`** (SSE → URL query; WS → first socket message) and is checked **before any handler runs**; an Origin/Host allowlist rejects wrong-origin callers. Bad token ⇒ `UNAUTHORIZED`.
3. **SSE flavour** (`server-sse.mjs` + `test-sse.mjs`): `createHTTPServer` (standalone) + client `splitLink`(subscription→`httpSubscriptionLink`, else `httpBatchLink`). Node lacks a global `EventSource`, so the `eventsource` ponyfill is passed (the Chromium renderer has it natively).
4. **WS flavour** (`server-ws.mjs` + `test-ws.mjs`): `ws` `WebSocketServer` + `applyWSSHandler` (keepAlive ping/pong) + client `createWSClient`(lazy, bounded `retryDelayMs`) + `wsLink`.
5. **A protocol-agnostic TCP “kill-proxy”** (`proxy.mjs`): the client connects to the proxy, the proxy pipes to the real server, and I can **hard-RESET the live sockets** (and, for the outage test, take the whole listener **down for 2.5 s**) — the most faithful “force a real network disconnect”, identical for SSE and WS.
6. **Two scenarios per transport**, asserting **every seq in `[0..N]` is delivered** (no drops), counting reconnects + at-least-once seam dups: (a) **mid-stream reset** at seq 8 (`scenario.mjs`); (b) **2.5 s full outage** at seq 10 while the server keeps emitting (`test-outage.mjs`). Plus a **negative-auth** probe (bad token must error pre-handler).

Commands (hard-timeout-wrapped): `npm install @trpc/server@^11 @trpc/client@^11 zod@^3 ws@^8 eventsource@^3` · `node test-sse.mjs` · `node test-ws.mjs` · `node test-outage.mjs sse` · `node test-outage.mjs ws`. Pinned: **@trpc/server + @trpc/client 11.18.0**, ws 8.21.0, zod 3.25.76, eventsource 3.0.7. API confirmed current via Context7 (`/trpc/trpc`: `tracked()`/`lastEventId`, `httpSubscriptionLink`, `wsLink`/`createWSClient`, `connectionParams`).

## Evidence

**(1) SSE — auth rejected pre-handler, lossless across a forced mid-stream reset** (`test-sse.mjs`):
```
[sse] node EventSource global present=false; using ponyfill
[sse-neg] rejected=true detail=UNAUTHORIZED
[sse] forced disconnect at seq=8 (reset 4 sockets)
[sse] delivered=31 unique=31 range=0..30 reconnects~=2 dups(at-least-once seam)=0 missing=[]
[sse] RESULT PASS (auth-rejected=true, lossless-resume=true, reconnects~=2)
```

**(2) WS — same router, same guarantees** (`test-ws.mjs`):
```
[ws-neg] rejected=true detail=UNAUTHORIZED
[ws] forced disconnect at seq=8 (reset 2 sockets)
[ws] delivered=31 unique=31 range=0..30 reconnects~=1 dups(at-least-once seam)=0 missing=[]
[ws] RESULT PASS (auth-rejected=true, lossless-resume=true, reconnects~=1)
```

**(3) The hard one — a 2.5 s FULL outage (listener gone, ~62 events pile up server-side), both transports resume losslessly** (`test-outage.mjs`):
```
[sse-outage] proxy DOWN at seq=10 (server head ~11)
[sse-outage] proxy UP again at server head ~73 (gap ~62 events during outage)
[sse-outage] unique=61 range=0..60 reconnects~=2 missing=[]
[sse-outage] RESULT PASS (lossless across ~2.5s full outage)

[ws-outage] proxy DOWN at seq=10 (server head ~11)
[ws-outage] proxy UP again at server head ~72 (gap ~61 events during outage)
[ws-outage] unique=61 range=0..60 reconnects~=9 missing=[]
[ws-outage] RESULT PASS (lossless across ~2.5s full outage)
```

**Findings.**
- **Both work; both are lossless.** `tracked()` + `input.lastEventId` + a bounded **server-side replay buffer** is what guarantees no-drop — *not* the transport. The lossless contract lives in the (transport-agnostic) router; I ran the **identical `appRouter` under both adapters**. ⇒ the decision is **cheaply reversible** (swap adapter + link, keep the router).
- **Reconnect cadence differs.** SSE/EventSource honours the **server-suggested retry** interval (~2 attempts across 2.5 s); WS `createWSClient` uses **your `retryDelayMs`** (~9 attempts at 500 ms) — finer-grained, configurable, plus `broadcastReconnectNotification()` for a *graceful* worker-restart (maps to §16 supervision) and keepAlive ping/pong liveness (maps to System Health).
- **Auth fit is the real differentiator on loopback.** SSE `connectionParams` ride **in the URL** (native `EventSource` can’t set headers) → the per-launch token lands in a request line, leaning on §16 URL-redaction discipline (or an `EventSourcePolyfill` header dep in the renderer). WS `connectionParams` ride the **first socket message** → token **never in a URL**; Origin is natively on the upgrade request for the §5 allowlist. This directly honours safety rule 7 (“secrets never in logs”) without extra redaction load.
- **SSE’s one structural advantage — proxy/firewall traversal — is MOOT here.** The §10 link is **127.0.0.1 loopback only** (§1/§4/§13); there are no intermediaries to befriend, so WS’s upgrade handshake costs nothing.
- **Backpressure caveat (applies to BOTH):** tRPC `onData` is push (no consumer-driven backpressure at the callback); transport flow control sits between the server generator and the socket, and the server’s `events.on(ee,…)` async-iterable **buffers unbounded** if the generator stalls. Bound it with a **replay window + `maxBufferedItems`**; the replay window is also the resume horizon (events older than the window can’t be replayed → consumers must tolerate a “resync from snapshot” signal).
- **At-least-once:** 0 dups observed, but a reconnect *can* re-deliver the in-flight event → §10 consumers (esp. approval updates) must be **idempotent by event id**, consistent with the architecture’s idempotency posture.

## Decision: GO-WITH-CONDITIONS — **tRPC subscription over WebSocket** (`wsLink` + `createWSClient` + `applyWSSHandler`), with `tracked()`/`lastEventId` + a bounded server replay buffer.

Rationale, specific to *this* architecture (not a generic SSE-vs-WS verdict):
1. **Best §5/§16 auth + redaction fit** — the per-launch session token is delivered **off-URL** (first WS message), so it never reaches a loggable request line; Origin is native on the handshake. No renderer `EventSource` polyfill needed (Chromium has native `WebSocket`).
2. **SSE’s only edge (proxy-friendliness) does not exist on loopback** — so choosing WS forfeits nothing.
3. **Reconnection + backpressure proven equivalent and lossless** (mid-stream reset *and* 2.5 s outage, 0 missing) — and WS adds configurable retry, graceful `broadcastReconnectNotification()` on supervised worker restart (§16), and keepAlive liveness for System Health.
4. **Reversible** — the lossless contract is in the transport-agnostic router; SSE is a drop-in fallback.

Why **GO-WITH-CONDITIONS** not bare GO: the primitive is firmly decided and its mechanics are proven with real sockets, but the load-bearing path was exercised with a Node `ws` client + a kill-proxy, **not yet inside the real Electron renderer** (native `WebSocket`, preload-injected token, real renderer Origin, supervised-restart reconnect). That wiring is the condition (Live-validation gate) — a downstream implementation step, not a missing external asset.

## No-go branch

The plan treats WS/SSE as interchangeable for §10, so the fallback is concrete and **validated in this spike**: **SSE via `httpSubscriptionLink`**. Because the server router (`tracked()` + `lastEventId` + replay buffer) is identical, switching is: mount `createHTTPServer` instead of `applyWSSHandler`, and client-side `splitLink`(subscription→`httpSubscriptionLink`, queries/mutations→`httpBatchLink`). Token moves to URL `connectionParams` (native `EventSource`) **or** to a header via `EventSourcePolyfill`. Adopt SSE if a future requirement reintroduces a proxy/firewall hop (e.g. the V1.1 hosted control plane behind the `EventIngressAdapter` seam) or if avoiding a `ws` server dependency becomes desirable. Both flavours passed identical reconnection/no-drop tests above.

## Pinned

- **Primitive:** tRPC v11 **subscription over WebSocket** — server `applyWSSHandler` (`@trpc/server/adapters/ws`) over `ws` `WebSocketServer`; client `createWSClient` + `wsLink` (`@trpc/client`).
- **Commands/queries:** `httpBatchLink` over loopback HTTP (token via header), routed with `splitLink` (subscriptions→`wsLink`); matches §10 “tRPC for commands/queries; a single push stream for status.”
- **Resumability contract:** every event = `tracked(String(monotonicEventId), uiSafeProjection)`; subscription input `{ lastEventId }`; **bounded server-side replay buffer** keyed by event id; consumers **idempotent by event id** (at-least-once).
- **Auth (§5):** per-launch session token via `connectionParams` (first WS message) → validated in `createContext` **before** the handler; Origin/Host allowlist from the upgrade request; reject `UNAUTHORIZED`/`FORBIDDEN` pre-handler. Loopback bind only.
- **Liveness/reconnect:** `keepAlive: { pingMs: 1000, pongWaitMs: 2000 }`; client `lazy` + bounded `retryDelayMs` (recommend capped exponential backoff); `handler.broadcastReconnectNotification()` on graceful worker restart (§16).
- **Backpressure guard:** cap the server async-iterable with a replay window + `maxBufferedItems`; window size = resume horizon; over-horizon ⇒ emit a resync-from-snapshot signal.
- **Event payloads:** UI-safe projections only — **no secrets / no raw content** (§10 / §16 redaction; the Phase-1.1 event-name catalog is the const-union source of truth).
- **Versions:** @trpc/server 11.18.0, @trpc/client 11.18.0, ws 8.21.0, zod 3.25.76 · (SSE fallback only: eventsource 3.0.7 ponyfill, Node-side; renderer uses native) · Node 22.13.0.
