# Spike 0.2 — GBrain round-trip + version-pin   (status: GO-WITH-CONDITIONS)

> **One line:** GBrain ships **read-only / index-only** in V1 — the architecture's safe default that already satisfies DoD (§6/§14.5). The four §19 round-trip GO conditions (which would *unlock GBrain write-through*) cannot be proven in Phase 0 (GBrain not installed, no embedding key, and the KnowledgeWriter they test is a Phase-4 deliverable), so write-through stays OFF behind a **live-validation gate**. The pin file format + capture procedure are decided; the binding SHA is a documented placeholder pending that live run.

## Question / unknown

**OQ-006** (ARCHITECTURE.md §18) — what is the known-good GBrain SHA, and can GBrain safely participate in a **write-through round-trip** (DB-originated semantic facts becoming canonical Markdown) without becoming a "hidden brain"?

The architecture (§6) makes this a gated decision. GBrain is **derived** from canonical Markdown and read/query-only at the runtime MCP boundary (REQ-F-019 / KN-2). Write-through is only permissible if the **§19 round-trip GO conditions** hold:

1. **fs-watch audit** shows **zero non-KnowledgeWriter** Markdown mutations (no hidden brain — KN-4/KN-9).
2. A **GBrain index job concurrent with a KnowledgeWriter write loses no update** (LIFE-6 ordering + revision-id idempotency).
3. An **injected DB-only fact is flagged by the parity check** (parity defect → quarantine + System Health; §6, HealthItem `failureClass=parity_defect`).
4. **Markdown stays syntactically valid + semantically lossless** across the round-trip (REQ-D-001 — index rebuildable from Markdown).

Plus: **pin the known-good GBrain SHA → `config/gbrain.pin`** (§13 version-pin & upgrade gate; consumed by the §4.7 startup version-pin check).

**Spec anchors:** §6 (Knowledge: Markdown/Obsidian/GBrain/GCL) · §18 OQ-006 · §13 (GBrain version-pin & upgrade gate) · §14.5 (read-only/index-only no-go branch still meets DoD) · §4.7/§4.4/§4.8 (Phase-4 consumers). NO-GO ⇒ GBrain ships READ-ONLY/index-only.

## Decision: GO-WITH-CONDITIONS

**Recommendation: GO-WITH-CONDITIONS.**

What is **decided now (GO)** and unblocks Phase 4 immediately:

- **GBrain = `garrytan/gbrain`** (PGLite engine); pin lives in `config/gbrain.pin` with a parseable, documented format consumed by the §4.7 startup check.
- **V1 posture = read-only / index-only** — the architecture's explicit default (§6: *"If write-through containment cannot be proven in the Phase-0 GBrain spike, V1 ships GBrain read-only/index-only and still satisfies DoD"*; §14.5). The runtime MCP surface stays search/think/graph/timeline/schema-read/health only; no mutate path. DB-only semantic facts remain **parity defects** (quarantined), never a write path.
- The four GO-condition **checks are mechanically sound** — proven by the harness (13/13, fails-closed on each violation). The Phase-4 parity/reconcile code can adopt this exact methodology.

The **conditions** (must clear the live-validation gate before any change):

- **Write-through stays OFF** until the four §19 conditions are run **green** against the pinned GBrain build **and** the Phase-4 KnowledgeWriter. This is the §13 pin-bump gate verbatim: bump the SHA / enable write-through *only after* the §12 GBrain compatibility/parity suite is green **and** a full re-index from Markdown succeeds.
- **The binding SHA is not yet pinned** — `gbrain_sha = PENDING_LIVE_VALIDATION` (the §4.7 check fail-closes on the sentinel → GBrain disabled/degraded until filled). Candidate `08b3698…` (gstack's v0.18.2 known-good) is the recommended starting build for the capture procedure but is **not** binding until validated here.

**Rationale.** A clean binary GO is impossible to assert honestly at Phase 0, and a NO-GO would mislabel the *safe default* as a failure. GO-WITH-CONDITIONS captures the truth: the read-only architecture is GO and ready to build; the *write-through extension* is deferred behind a concrete, named gate. Critically, **Phase 4 is not blocked** — read-only/index-only is the build target regardless of how the gate later resolves.

## No-go branch

The plan's named no-go (**"GBrain ships READ-ONLY/index-only, still satisfies DoD per §6/§14.5"**) is **already the chosen V1 posture** — so there is no separate fallback to invoke; the conservative branch *is* the baseline. Made concrete:

- Runtime MCP exposes only read/query capabilities (§4.7); the adapter never opens the PGLite file directly (single-owner per-workspace process, §13).
- Every semantic mutation routes through a validated `KnowledgeMutationPlan` → KnowledgeWriter → Markdown; GBrain is rebuilt **from** Markdown (REQ-D-001). Full re-index recovers every node, so losing the index is never a data-loss event.
- If the live gate later **fails** (write-through containment unprovable on the real engine), nothing changes — V1 stays here and meets DoD. If it **passes**, write-through can be enabled as a post-V1 increment without re-architecting.
