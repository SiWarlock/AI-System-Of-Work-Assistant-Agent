## Evidence

### Cost-cap anchor arithmetic (the only number that needed live data)

The most expensive routine job is **`meeting.close`**: a transcript + structured extraction. Sizing a 60-min meeting ≈ 9,000 words ≈ ~12k tokens of transcript (Opus-4.7+ tokenizer runs ~1–1.35×, so 12–16k) + ~2–3k system/output-schema prompt ⇒ **~15k input**, ~**2k output** structured extraction.

```
Single-shot meeting.close (ModelProviderPort, raw extraction):
  Opus 4.8 : 15k×$5/1M  + 2k×$25/1M = $0.075 + $0.050 = $0.125
  Sonnet 4.6: 15k×$3/1M + 2k×$15/1M = $0.045 + $0.030 = $0.075
  Haiku 4.5 : 15k×$1/1M + 2k×$5/1M  = $0.015 + $0.010 = $0.025

Agentic meeting.close (AgentRuntimePort, ~5 turns, UNCACHED worst case
  ≈ 100k cumulative input / 10k output on Opus 4.8):
  100k×$5/1M + 10k×$25/1M = $0.50 + $0.25 = $0.75
  (prompt-caching the repeated transcript at 0.1× reads drops this to ~$0.20–0.35)
```

⇒ A **$1.00** default cap on `meeting.close` never false-trips a legitimate job (single-shot $0.075–0.125; agentic worst-case $0.75) yet **halts a runaway agent loop within ~$1 of waste** — exactly the COST-1/COST-2 intent (cancel-with-no-partial-side-effect, `ARCHITECTURE.md §7`, task `5.4`).

**Rationale per row:**

- **D1 — meeting-closeout E2E (`§9` flow 1).** Background **async** workflow (Granola poll → correlate → `meeting.close` AgentJob → validator → KnowledgeWriter → Tool Gateway → GBrain re-index → dashboard), **not** a user-facing spinner. Dominated by the `meeting.close` LLM extraction, which is itself bounded by its `maxRuntimeSeconds` default (180 s, below). Its tail segments (KW→GBrain ≤60 s, KW→read-model ≤10 s) are **already hard-gated (H2/H3)** and partly overlap, so a single hard E2E gate would double-count and is sensitive to provider class (cloud vs local) + transcript length — neither pinned until spike 0.5. Comparable async meeting-AI systems (Granola/Otter/Fireflies) surface notes in ~30–90 s. **180 s** gives generous headroom with most mass < 2 min; **local routes** get a looser 600 s SLO because Mac-local inference throughput is the variable. **Poll cadence is excluded** — it is a separate config-default (see Pinned), not part of this latency.
- **D2 — Copilot (`§9` flow 13, read-only).** Interactive RAG: workspace-scoped (or GCL Visibility Gate) retrieval + schema-gated cited synthesis, streamed over the `§10` push channel (WS/SSE chosen in spike 0.4). Anchored to interactive-chat UX: **TTFT < 3 s** keeps a streamed answer feeling alive; **full ≤ 20 s** stays under Nielsen's 10-s-attention threshold-with-progress band that Perplexity/ChatGPT cited answers occupy. No side effects.
- **D3 — Approval (`§9` flow 8).** **Human think-time is unbounded by design** (deferred-approval snooze default 24 h, expiry 7 d) — so the budget is **system-side latency only**. The interactive part is the exactly-once compare-and-set transition + push to **both** Mac + Telegram surfaces: **≤ 2 s** (matches the <2 s dashboard feel; the atomic CAS in task `5.x`/`8.x`). The downstream external write through the Tool Gateway envelope completes **≤ 10 s** on a healthy connector (bounded by the external API; a failure routes to the retry/outbox and is **not** counted against the budget).
- **D4 — ingress→workflow-start.** Local control-plane plumbing: `EventIngressAdapter` accept → operational-store persist → **workspace assignment (REQ-F-002, deterministic)** → local Temporal `client.start()` (dev server, tens of ms). **≤ 2 s** absorbs durable persistence + assignment. **NB:** this is distinct from perceived end-to-end latency, which is dominated by **poll cadence** (a separate config-default).
- **D5 — concurrency cap.** A single Mac runs Electron + worker + Temporal dev-server + **one GBrain sidecar per workspace** (≥3). LLM calls dominate cost and **local inference serializes on one GPU**, so the AgentJob layer is throttled deliberately rather than using the Temporal SDK defaults (~100). **4** concurrent cloud LLM AgentJobs / **1** for any route pinned to a local provider; **16** activity slots / **32** workflow-task slots for cheap orchestration. Built configurable; the precise tuning is a measured Phase-7/12 concern.
- **D6 — default `maxCostUsd` (OQ-004, COST-2).** Guardrail, not a budget — sized against the most-expensive plausible route per the arithmetic above. Per-capability table in Pinned. **Local-provider routes have $0 marginal cost**, so the cap is effectively N/A there and `maxRuntimeSeconds` is the binding governor.

## Decision: GO

GO — all six deferred budgets are now **resolved**: four **SET as advisory SLOs** (D1–D4), two **SET as config defaults** (D5–D6), with explicit rationale anchored to the architecture and to comparable systems; the three PRD budgets are **restated as the only hard DoD gates**. `OQ-004` is closed (default `maxCostUsd` **and** `maxRuntimeSeconds` per capability pinned below). Nothing here changes a contract surface — the `AgentJob` already carries `maxRuntimeSeconds` (required) + `maxCostUsd?` (optional) per task `1.6`; these are *default values* the Broker stamps onto uncapped jobs (`5.4`), tunable at any time. The advisory SLOs are deliberately **not** wired as failing assertions — only H1–H3 are. This matches the plan's design: the perf pass *sets* the soft targets here and *measures* them in Phase 7/12.

## No-go branch

The plan's named fallback is **"mark deferred to a measured pass in Phase 7/12 with why."** That fallback is **partially exercised by design**: the SLO **values** are set now (so Phase-7 workflows have a target to build against and System-Health/OBS thresholds to alarm on), but their **enforcement remains deferred to measurement** — if a Phase-7/12 measured pass shows a target is systematically unachievable on real hardware with the chosen provider, the number is re-set there (the SLOs are advisory, so this is a config edit, not a contract or DoD change). The three hard gates have no no-go branch — they are immovable PRD requirements; if H2/H3 cannot be met, that is a §19 GBrain-round-trip / storage-pipeline finding, not a budget relitigation.
