# Spike 0.3 — Hermes Adapter Surface   (status: GO-WITH-CONDITIONS)

## Question / unknown

**OQ-007 (ARCHITECTURE.md §7, §18):** Which Hermes integration surface should the
`HermesRuntimeAdapter` (the `AgentRuntimePort` RT-1 adapter) use — **API server / TUI gateway /
Python (CLI) wrapper / Kanban / hybrid** — judged by a bounded meeting-close mock exercising:
schema-validated structured output, stop/cancel, log capture, controlled-tool gating, and install
friction?

`AgentRuntimePort` contract (§7): agentic runtimes that carry tool-policy / MCP / structured-output /
subagent semantics, run `meeting.close` AgentJobs, and obey the **strict side-effect rule** —
`runtime output → schema gate → validator → KnowledgeMutationPlan/ProposedAction → KnowledgeWriter/Tool Gateway`
(no runtime output reaches a write adapter directly). NO-GO is explicitly **non-blocking**: the Claude
Agent SDK runtime (or any conformance-passing runtime) may carry the `meeting.close` critical path
(§7 owner-finalization decision); Hermes stays required + DoD-tested (§12) but **not** first-install
gating (REQ-I-002/003, REQ-NF-005 clean-install-without-Hermes).

## Evidence

**1. Structured output via `-Q` is clean + parseable (real run, exit 0, 9s):**
```
{"meeting_title": "Parser Refactor Ship Decision",
 "decisions": ["Ship the parser refactor this week", "Drop the legacy CSV path"],
 "action_items": [{"owner": "Alex", "task": "Ship the parser refactor", "due": "this week"},
                  {"owner": "TBD",  "task": "Update the docs",           "due": "TBD"}]}
```
`-Q` (quiet/programmatic) emitted ONLY the final JSON on stdout; `session_id` went to stderr. No banner, no spinner.

**2. Schema gate PASS on the real output (the §7 candidate-data gate):**
```
JSON.parse: OK
manual structural gate: PASS (jsonschema lib absent)
no-inference (REQ-F-017): 1 action item(s) correctly owner=TBD
```
The unassigned "Update the docs" item came back `owner:"TBD", due:"TBD"` — the no-inference behavior
held even before our validator (which enforces it regardless).

**3. Controlled-tool gating works and is observable.** `-t clarify` restricted the run to exactly
**"1 tools · 0 skills"**, banner `clarify: clarify`, `Skills toolset disabled`. Source confirms the
mechanics + a sharp caveat: `_normalize_toolsets("")` returns `None` → `use_config_toolsets=True` →
**an empty/omitted `-t` falls back to the user's full config toolset (incl. mutating tools)**. Read-only
gating therefore requires an **explicit** minimal toolset, never empty.

**4. Stop/cancel + budget enforcement leaves no partial output:**
```
exit_code=124 (124 = SIGTERM by timeout)  elapsed=4s
stdout_bytes=0  (no final output => nothing reaches schema gate => no side effect)
no lingering hermes chat process (clean kill)
```
Killing the subprocess mid-inference = a clean budget/cancel: zero stdout → nothing crosses the schema
gate → no `KnowledgeMutationPlan`, no write (COST-1 "cancel with no partial side effect").

**5. Kanban is a durable, JSON-observable board (real, free):**
```
hermes kanban create --json → {"id":"t_4f1b03ca","status":"ready","result":null,
   "session_id":null,"max_retries":null,"created_by":"user","created_at":1782786684,...}
```
SQLite-backed, atomic claim, dependencies, `runs`/`log`/`tail`/`stats` all `--json`. BUT: *"Without a
running gateway, tasks stay in 'ready' forever"* — Kanban needs the **gateway dispatcher daemon**
running to execute. It is a fire-and-forget durable queue, **not** a synchronous request/response runtime.

**6. ACP available:** `hermes acp --check` → `Hermes ACP check OK` (JSON-RPC server for editor
integration; native cancel + tool-permission events; but a long-running stdio session daemon).

**Surface scorecard** (evidence-backed; ✔ = verified live this spike):

| Axis | one-shot CLI (`chat -q`) | ACP server | Kanban | proxy | mcp serve | TUI gateway |
|---|---|---|---|---|---|---|
| schema-validated output | prompt + downstream gate ✔ | JSON-RPC blocks | task `result --json` ✔ | raw model only | tool-result | poor |
| stop/cancel | process kill, exit124 ✔ | native cancel notif | reclaim/heartbeat | n/a | n/a | n/a |
| log capture | stdout/stderr ✔ | JSON-RPC stream | runs/log/tail ✔ | proxy log | mcp log | chat log |

## Decision: GO-WITH-CONDITIONS

**Recommended surface — HYBRID:**
- **Primary (synchronous `meeting.close` AgentJobs): one-shot CLI subprocess.**
  `hermes chat -q <prompt> -Q -t <readonly-toolset> -m <model> --provider <p> --max-turns <N> [--safe-mode]`,
  spawned per job. It cleanly satisfies all five axes with the **lowest** lifecycle friction (no daemon),
  and the per-job subprocess maps 1:1 onto the AgentJob request→validated-output→done shape. Stop/cancel =
  process kill (proven), structured output = prompt + our ajv gate (the §7 candidate-data model; proven),
  tool gating = `-t` whitelist (proven), logs = captured stdout/stderr → §16 redaction layer.
- **Optional (the §9 RT-7 Hermes-autonomous-automation path): Kanban.** Durable SQLite board with `--json`
  observability for owner-defined automations — but every external write still routes through the Tool
  Gateway and every semantic write through KnowledgeWriter (one-writer invariant; §9). Temporal stays the
  product-workflow source of truth; Kanban is not.

**Rejected:** TUI/messaging `gateway` (human-facing, wrong shape) · `proxy` (that's a *ModelProviderPort*
raw-model surface, no tool loop — not `AgentRuntimePort`) · `mcp serve` (inverted: exposes Hermes *as a
tool to* another agent, not a runtime executing our AgentJobs). **ACP** is a viable structured alternative
(JSON-RPC, native cancel + permission events) kept as the **fallback** if we later need streaming
tool-permission gating — but its long-running stdio-session daemon is heavier than per-job subprocesses.

**Conditions (must hold in the §12 conformance suite before the adapter is matrix-eligible):**
1. The read-only `meeting.close` toolset is an **explicit minimal set** (start: `clarify` only) — never
   empty `-t` (empty falls back to config tools, incl. mutating). Conformance asserts the launched run
   reports **0 mutating tools**.
2. Untrusted-transcript (ING-7) runs launch with `--safe-mode` (or `--ignore-user-config --ignore-rules`)
   to strip injected AGENTS.md / memory / MCP, so untrusted content cannot reach a configured mutating MCP.
   (Note: `--ignore-rules` is a guard-disarming flag — used only inside the controlled adapter, gated by
   the ING-7 admission test; not a casual shell flag.)
3. Hermes output is **candidate data** — always ajv-gated + validated downstream; never trusted directly
   (§7 strict side-effect rule; proven necessary — output is free-form text, no native JSON-schema mode).
4. Hermes stays **not first-install-gating**: adapter is spawn-if-present; absent → `HermesRuntimeAdapter`
   reports unavailable and the ProviderMatrix routes `meeting.close` to Claude SDK / another conformant
   runtime (REQ-NF-005).

## No-go branch

Already concrete and **non-blocking**: if the conformance conditions can't be met (tool gating proves
unreliable, or subprocess lifecycle is flaky), the **Claude Agent SDK runtime carries the `meeting.close`
critical path** (§7 owner-finalization decision — the matrix may route to any conformance-passing runtime).
`HermesRuntimeAdapter` remains a **required, DoD-tested** adapter (RT-1, §12) but **not** first-install
gating (REQ-I-002/003, REQ-NF-005). The clean-install acceptance test runs on the default provider runtime
**without Hermes** regardless of this spike's outcome.

## Pinned

- **Hermes:** `0.17.0 (2026.6.19)`, upstream `fbf748b2`, Python `3.11.15`, at `~/.hermes/hermes-agent`.
- **Primary surface:** one-shot CLI subprocess — `hermes chat -q <prompt> -Q -t <readonly> -m <model> --provider <p> --max-turns <N> [--safe-mode]`.
- **RT-7 autonomous surface:** Kanban (`hermes kanban …`, SQLite board, `--json` on create/list/show/runs/log/stats/dispatch); requires `hermes gateway` dispatcher running.
- **Structured fallback surface:** ACP (`hermes acp`; deps verified `OK`).
- **Programmatic flags:** `-Q` (quiet/programmatic output), `-t` (explicit toolset whitelist), `--max-turns N` (turn cap → budget), `--provider`/`-m` (route), `--source tool` (hide from user session lists), `--safe-mode`/`--ignore-user-config`/`--ignore-rules` (untrusted-content isolation).
- **Read-only meeting.close toolset:** explicit minimal set, seeded at `clarify`; **never** empty `-t`.
- **Cancel/budget mechanism:** subprocess SIGTERM (wrap each job in a runtime-seconds timeout); pre-final-output kill ⇒ no side effect.
