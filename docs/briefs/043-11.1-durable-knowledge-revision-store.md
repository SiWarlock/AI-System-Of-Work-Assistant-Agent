# /tdd brief — durable_knowledge_revision_store

## Feature
Build the **durable** `KnowledgeRevisionStore` — a new operational-store repository (Drizzle table + SQLite/Postgres adapters + a store adapter) that persists `CommittedRevision` records keyed by the KnowledgeWriter's `idempotencyKey`, so idempotent-replay of a semantic commit survives a worker restart. Today the store is an in-memory `Map` (an `arch_gap` — no durable impl exists); this replaces it with real persistence. ## Acceptance criteria (what "done" means)
- [ ] A new `KnowledgeRevisionRepository` (db-layer) persists a `CommittedRevision` and retrieves it by `idempotencyKey`; a novel key records, a seen key returns the prior record (idempotent).
- [ ] The `idempotencyKey` column carries a **UNIQUE** constraint; a duplicate `record` for the same key is idempotent (first-write-wins / on-conflict-no-op) — NEVER two revisions for one key (the exactly-once substrate).
- [ ] The repo passes the ONE `repository-contract.test.ts` suite on **BOTH SQLite AND Postgres** (no dialect-specific SQL; Postgres is never a permanent stub — the forbidden pattern).
- [ ] An **additive migration** creates the table (pre-migration backup + app↔schema compat rule; no destructive change).
- [ ] A `KnowledgeRevisionStore` adapter over the repo satisfies the `packages/knowledge` interface (`getByIdempotencyKey` / `record`), so KnowledgeWriter's `applyPlan` binds to it unchanged.
- [ ] **Idempotent across restart** — a FRESH store/adapter instance over the SAME db sees a prior instance's recorded revision (the in-memory `Map` fails this; the durable repo passes). This is the load-bearing property.
- [ ] All new unit + contract tests pass; repo-wide `pnpm -w turbo run typecheck test` green (+ `SOW_PG_DOCKER` for the Postgres contract leg if the harness needs it); `/preflight` clean. **MANDATORY Step-8 review = security-reviewer + code-quality-reviewer** (the durable exactly-once substrate for the sole writer).
- [ ] No hard line crossed — local operational-store persistence only; not the propose bridge, not GBrain write-through, no external I/O.

## RED test outline (Step 2)
1. **`revision_repo_record_and_get_by_idempotency_key`** — Asserts: record a `CommittedRevision`, `getByIdempotencyKey` returns it; unknown key → undefined. Why: the store contract.
2. **`revision_repo_idempotency_key_unique`** — Asserts: a second `record` for the same `idempotencyKey` does NOT create a second row (first-write-wins / no-op); still exactly one revision for the key. Why: exactly-once substrate (§16).
3. **`revision_repo_contract_both_dialects`** — Asserts: the repo passes the shared contract cases on SQLite AND Postgres (parametrized like the existing repos). Why: no dialect-specific SQL (forbidden pattern / §4).
4. **`revision_store_adapter_satisfies_interface`** — Asserts: the `KnowledgeRevisionStore` adapter over the repo returns `CommittedRevision | undefined` from `getByIdempotencyKey` and persists via `record`. Why: KnowledgeWriter binds to it unchanged.
5. **`revision_store_idempotent_across_restart`** — Asserts: instance A records a revision; a FRESH instance B over the SAME db returns it via `getByIdempotencyKey` (simulated restart). Why: the load-bearing durability property (the in-memory `Map` fails this).
6. **`revision_migration_additive`** — Asserts: the migration creates the table without a destructive change; app↔schema compat holds. Why: migration-safety rule.
