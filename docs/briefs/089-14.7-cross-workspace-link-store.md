# /tdd brief — cross_workspace_link_store (14.7 worker leg)

## Feature
The `CrossWorkspaceLink` entity + durable dual-dialect store + an owner-approval flow (create/approve/revoke) as the **single sanctioned WS-8 cross-read input**, feeding APPROVED links into the GCL cross-workspace read path so a coordination/global read blends **exactly the approved, sanitized slice of workspace B into workspace A — and nothing else**. Absent an approved link, **ZERO raw cross-workspace content surfaces** (WS-8 stays fully isolating). Worker/db-track; the owner-approval UI surface (`surfaces/links/`) is a later desktop-impl slice. **This is the highest-bar safety slice of Phase 14 — safety rule 4.**

## Use case + traceability
- **Task ID:** 14.7 (worker leg)
- **Architecture sections it implements:** `ARCHITECTURE.md §19.1` (primary), `§5` (WS-8 isolation root; the cross-workspace link is the single sanctioned WS-8 exception), `§6` (GCL Visibility Gate consumes approved links). **Implements safety rule 4** (the approved link is the ONLY cross-workspace read path).
- **Related context (confirm at Step 1):**
  - **The GCL read path already exists:** `GclProjection` (`packages/contracts/src/models/gcl-projection.ts`) is the SINGLE cross-workspace read unit (REQ-F-005/WS-8) — sanitized, visibility-scoped, NEVER raw content; its leakage shape-gate is KEY-NAME-INDEPENDENT (rejects any multi-line / >1024-char string value recursively, regardless of key). The global surface read-model key is `READ_MODEL_KEYS.global = "global_surface"` (workspaceId=null). There is **no** `visibility-gate.ts` — the "gate" is the GclProjection sanitizer + the `queries.ts` / `api/projections/uiSafe.ts` build path. **Confirm at Step 1 the exact global/coordination read producer** where approved-link consultation must gate which workspaces' slices blend in.
  - **Approval model is FROZEN:** `Approval.subjectKind` is a frozen 2-value enum `{external_action, semantic_mutation}` with a biconditional refine — **do NOT add a `cross_workspace_link` subjectKind** (that's a frozen-Appendix-A-seam change / contract track). The link carries its OWN `status` (pending|approved|revoked); the approve/revoke flow is dedicated (Step-2.5 Q1).
  - **Pattern:** the 14.1/14.6/14.2 operational-record pattern (db-owned Row DTO + repo-contract both dialects + WS-8 membership gate + the immutable workspace-binding anchor — **worker Lesson 30**); the `resolveKnownWorkspace` registry membership gate.

## Acceptance criteria (what "done" means)
- [ ] A `@sow/db` cross-workspace-link table + dual-dialect (SQLite + Postgres) migration + a `CrossWorkspaceLinkRepository` (`create`, `get`, `listApprovedForReader(workspaceId)`, `setStatus`) passing the ONE repo-contract suite on BOTH dialects.
- [ ] The `CrossWorkspaceLink` shape: `{ linkId, fromWorkspaceId (the READER, A), toWorkspaceId (the SOURCE, B), scope (a BOUNDED selector of B's slice — e.g. projectionType/visibilityLevel), status (pending|approved|revoked), createdAt, approvedAt?, revokedAt? }`. Both workspaces must be REGISTERED (14.1 membership gate) at create.
- [ ] `crossWorkspaceLink.ts` procedure — `create` (mints a `pending` link, both ws registered), `approve` (owner: pending→approved), `revoke` (→revoked); boot-wired, auth-gated, redaction-safe (§16).
- [ ] **THE safety boundary (security-reviewer=invariant, safety rule 4) — absent-link ⇒ zero bleed:** with workspace B seeded with real content, a cross-workspace/coordination read for workspace A with NO approved A→B link surfaces **ZERO** of B's content (raw OR sanitized). NON-VACUOUS: seed B, assert the read is empty of B.
- [ ] **Approved link ⇒ ONLY the sanitized approved slice:** an approved A→B link makes B's slice readable from A **only through the GclProjection sanitizer** (never raw bytes — the leakage shape-gate still applies) and **only the link's `scope`** (nothing outside it). A test proves a raw-content-shaped value in B does NOT cross even with an approved link.
- [ ] **Revoke closes the path:** revoking an approved link returns A's cross-workspace read to zero-of-B (immediately). A test drives approve→read(slice visible)→revoke→read(empty).
- [ ] **WS-8 direction + immutability:** the read authorization is directional (an approved A→B link authorizes A reading B's slice, NOT B reading A); `fromWorkspaceId`/`toWorkspaceId` come from the stored link, never a caller field. Per Lesson 30, a link's `(fromWorkspaceId, toWorkspaceId)` pair is **immutable** — re-creating a link id with a changed pair is rejected (`cross_workspace_link_binding_immutable`); to change scope/pair, revoke + create anew.
- [ ] `/preflight` clean (worker + repo-wide; repo-contract both dialects).

**THE WS-8 safety boundary (the crux — non-vacuous):**
6. `absent_link_zero_cross_workspace_bleed` — B seeded with content; A's cross-workspace read with NO approved A→B link ⇒ ZERO of B surfaces. Why: **safety rule 4** (default isolation).
7. `approved_link_surfaces_only_sanitized_scoped_slice` — an approved A→B link surfaces B's slice ONLY through the GclProjection sanitizer AND only within `scope`; a raw-content-shaped value in B does NOT cross. Why: safety rule 4 + the leakage gate.
8. `revoke_closes_the_read_path` — approve→read(visible)→revoke→read(empty). Why: revocation is immediate.
9. `link_direction_is_one_way` — an approved A→B link does NOT let B read A. Why: directional authorization (WS-8).
