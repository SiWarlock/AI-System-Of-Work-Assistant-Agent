# /tdd brief — parity_reconcile_composition

## Feature
Build the **pure, worker-side reconcile-composition helper** (piece **A** of the reconcile-TRIGGER arc, task 13.10) — the smallest-safe-first slice. A single composition function runs `reconcileParity(req, deps)` over **fully injected** inputs `(canonicalSet, dbProjection, optional rebuildOracle)`, calls the built B3 `recordReconcileOutcome` gate to persist the report, routes the reconcile pass's `outcome.healthItems` to an **injected health sink**, and returns the `ParityRecordDisposition`. **No workflow, no boot, no real I/O, no Temporal, no gbrain** — every dependency is a fake in tests. Ships **DORMANT + reachability-waivered**: no production caller (the Temporal/scheduled trigger + the real DB-projection builder + the real gbrain read grant are later pieces B–F, two of them OWNER-GATED). This is the composition seam every later piece wires **into** — it defines the shape of "run one reconciliation pass" without owning where the pass is triggered or where its inputs come from.

## Acceptance criteria (what "done" means)
- [ ] A single exported composition helper (default name **`runReconcilePass`**) with signature `runReconcilePass(req: ReconcileRequest, deps: ReconcilePassDeps): Promise<ParityRecordDisposition>`, where `ReconcilePassDeps` injects: the `ReconcilerDeps` (id/clock generators), the `ParityReportRecorder` (B3 write-port), and the **health sink port** (shape per Step-2.5 #1). Every dep is a fake in tests.
- [ ] **Happy path (recorded):** on a successful `reconcileParity`, the helper calls `recordReconcileOutcome(outcome, recorder)` → `{kind:"recorded"; report}`, then routes **every** element of `outcome.healthItems` to the injected health sink **in order**, then returns the `recorded` disposition. Asserts: the reconciler ran once, the report persisted once, each health item was routed exactly once.
- [ ] **Clean pass, no health items:** a clean reconcile (`cleanForServing=true`, coverage complete, no divergence) records the report and routes **zero** health items — the sink is not called. Asserts: recorded disposition + 0 sink calls.
- [ ] **Dirty pass routes the defect:** a HARD-divergence reconcile (`cleanForServing=false`) records the (dirty) report verbatim AND routes the `parity_defect` health item. Asserts: recorded + the `parity_defect` item reached the sink (operational truth — the degrade signal is never dropped).
- [ ] **Skip (reconcile error) routes NOTHING to the sink + records nothing:** a `workspace_mismatch` / `report_invalid` reconcile `err` → the helper returns `{kind:"skipped_reconcile_error"; error}`, records nothing, and routes **zero** health items (the reconciler produced none on the err path). Asserts: skipped disposition + 0 recorder calls + 0 sink calls. (Whether the SKIP itself raises a health item is Step-2.5 #2 — default: NO, it's the trigger piece's call.)
- [ ] **Fail-closed — record fault propagates (fault ≠ skip):** a `record` store fault (the recorder rejects) propagates as a **rejection** of `runReconcilePass` — NOT swallowed, NOT downgraded to a skip; health items are **not** routed (the pass didn't durably land). Asserts: `runReconcilePass` rejects; 0 sink calls. Why: §12 fail-closed — a store fault must be visible (the caller degrades + raises a store-fault health item), distinguishable from a reconcile-error skip.
- [ ] **Fail-closed — health-sink fault propagates:** if the health sink faults while routing (rejects, or returns a typed `err` per its shape), the helper surfaces it (rejects / propagates) rather than silently dropping a trust-defect signal. Asserts: the fault is not swallowed. (Exact propagation shape rides Step-2.5 #1's port choice — a `HealthItem`-in `Promise<void>` port rejects; a `HealthSurface`-backed sink returns a typed `err` the helper must not ignore.)
- [ ] **Dormancy + reachability:** ships DORMANT + waivered — NO production caller, NO boot change, NO Temporal, NO real gbrain, NO real DB projection. Inputs are all injected; the two OWNER-GATED real bindings (the live `GbrainReadGrant` transport for the DB projection, the real `RebuildOracleSet`) stay **unbound**. `/wired` reports the helper reachable-by-test-only; the waiver names pieces B–F as the future wiring.
- [ ] All unit tests pass; repo-wide `pnpm -w turbo run typecheck test` green; `/preflight` clean.
- [ ] **MANDATORY Step-8 dual review = security-reviewer + code-quality-reviewer** — this composition feeds the serve-time trust gate (a wrongly-recorded clean report, or a dropped/mis-routed defect-health signal, is a trust-gate defeat). Safety-critical; own commit.

## Things to flag at Step 2.5
1. **[PRE-DELEGATED DECISION #3 — health-routing shape] How does the injected health sink consume `outcome.healthItems`?** The reconciler produces pre-built frozen `HealthItem`s (fresh `newHealthItemId()` per pass); the worker `HealthSurface.record` consumes a `HealthFailure` (needs `failureClass`/`subjectRef`/`message`/`auditRef`/`now`), and its OBS-2 dedupe keys on `healthItemDedupeKey(failureClass, subjectRef)` — **ignoring the item's id**, so it correctly de-dupes recurrences.
   - **(a) Narrow injected port `ReconcileHealthSink { record(item: HealthItem): Promise<void> }`** — piece A routes each `outcome.healthItems` element through it fail-closed; the REAL binding (surface-re-projection-for-OBS-2-dedupe vs direct-put) is decided + wired in piece D/F where the `HealthSurface` is actually in scope.
   - **(b) Re-project through `HealthSurface.record` NOW** — decompose each `HealthItem` back to a `HealthFailure` (requires a `subjectRef`, which the frozen `HealthItem` does not cleanly expose — a lossy/derived reconstruction), pulling the whole surface into this pure fakes-only slice.
   - **(c) Direct `HealthItemStore.put(item)` now** — no dedupe; since the reconciler mints a fresh id per pass, every reconcile pass would spawn a DUPLICATE health row (OBS-2 violation).
   - **My default vote: (a).** Keeps piece A pure + minimal + fakes-only per "smallest-safe-first"; the port shape (`HealthItem` in) exactly matches what the reconciler produces (no lossy decomposition); and it defers the surface-coupling to the piece where the surface lives. **I recommend the real binding re-project through the worker `HealthSurface` for OBS-2 dedupe** (a parity defect recurring every reconcile pass must bump `occurrenceCount` on ONE item, not spawn a duplicate per pass — the reconciler's fresh-id-per-pass makes id-keyed dedupe impossible), and I recommend piece D/F reconstruct the `HealthFailure.subjectRef` from a stable identity (workspace‖revision‖factIdentity) — this is the concrete sub-decision that rides the wiring piece, not piece A. Flag if you'd rather bind the surface now (option b) despite the purity cost.
2. **Does a `skipped_reconcile_error` (workspace_mismatch / report_invalid) also raise a health item?** The reconciler produces no `healthItems` on the err path. **Default vote: NO** — piece A returns the `skipped_reconcile_error` disposition and routes only the reconciler's `outcome.healthItems`; raising a health item FOR the skip is the TRIGGER piece's call (it holds the `origin` + retry context) — a named follow-up. Flag if you'd rather synthesize a skip health signal here.
3. **Return the bare `ParityRecordDisposition`, or augment it (routed-item count / `coverageComplete`)?** **Default vote: return the bare `ParityRecordDisposition`** (the Carry-forward spec says "return the `ParityRecordDisposition`"; on the `recorded` path the caller reads `report.coverageComplete`/`report.cleanForServing` directly). No augmentation.
4. **Helper name + file placement.** Default: **new file `apps/worker/src/composition/parityReconcile.ts`**, helper **`runReconcilePass`** — a distinct concern from `parityReportStore.ts` (which owns the store ports/adapters + the B3 record gate), and the natural home for pieces B/C's builders too. Flag if you'd rather co-locate in `parityReportStore.ts`.
