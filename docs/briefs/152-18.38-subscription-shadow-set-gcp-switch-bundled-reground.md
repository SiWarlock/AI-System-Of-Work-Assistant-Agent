## Feature
Extend the armed-subscription shadowing-env watched set (`SUBSCRIPTION_SHADOWING_ENV_KEYS`, **currently 30 keys**) to close the coverage gaps surfaced by a **⛔HARD repo-source re-ground against the SHIPPED claude-code binary** (v2.1.216 — NEWER than the guard's ~v2.0.x public-docs grounding). The load-bearing critical add is a **NEW 6th provider switch `CLAUDE_CODE_USE_ANTHROPIC_GOOGLE_CLOUD`** (the exact GCP analog of the covered `_USE_ANTHROPIC_AWS`): because it was UNWATCHED, the guard's generic-GCP-cred exclusion (`GOOGLE_APPLICATION_CREDENTIALS`/`GCLOUD_PROJECT`/…) is currently **FAIL-OPEN** — the precise worker-L72 soundness condition ("the generic-cred exclusion is sound ONLY when EVERY provider switch is watched"). Adding the switch **restores exclusion soundness** exactly as 18.37 did for the AWS switch. Alongside it: the `CLAUDE_CODE_USE_GATEWAY` redirect switch + a grounded cluster of new by-presence auth channels, egress-redirect base-URLs/proxies/socket, and an mTLS cert-store var. Because the constant is **single-sourced**, this hardens BOTH the `process.env` guard (`assertSubscriptionAuthEnv`) AND 18.36's settings-`env` leg in one edit. Over-inclusion is fail-safe (a spurious degrade only hits the owner-gated armed path; a miss is a silent fail-OPEN → include when uncertain, L65/L72). No behavior change to the shipped default — the guard runs ONLY on the armed boot path; extending the watched set is byte-equivalent when unarmed. Arms NOTHING; a §ARM-18 CHECKPOINT-1 pre-first-armed-run hardening.
