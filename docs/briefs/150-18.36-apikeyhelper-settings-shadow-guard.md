# /tdd brief — apikeyhelper_settings_shadow_guard

## Feature
Extend the armed-subscription no-surprise-spend defense to the ONE shadow the 18.28 env guard structurally cannot see: a **Claude-Code settings-level API-key injection**. The Agent SDK `query()` runs the Claude Code harness, which resolves credentials first-match-wins (`ANTHROPIC_API_KEY` → `ANTHROPIC_AUTH_TOKEN` → OAuth profile) and **also honors Claude Code `settings.json`** — so an `apiKeyHelper` (a command whose stdout becomes the key) or a settings-injected `env.ANTHROPIC_API_KEY` would make a "subscription" run silently authenticate with a **raw API key (real metered spend)**, invisible to the `process.env`-only shadowing guard. On the ARMED boot path, detect a settings-level key injection (PRESENCE only — never the value, rule 7) and **DEGRADE the arm fail-closed** (strip the transport gate → local/stub; never a worker crash, §16/L52). Default-OFF / byte-equivalent: the settings files are read ONLY on the armed path (`config.subscriptionArm?.enabled === true`); the shipped default never touches them. This slice arms NOTHING and closes a §ARM-18 CHECKPOINT-1 residual.

## Acceptance criteria (what "done" means)
- [ ] A **pure** guard resolves a settings-level-injection verdict from injected file-reads (fakeable reader; NO real fs/`os.homedir` in tests): given the Claude-Code settings hierarchy contents, it returns a typed fault when an injecting field is PRESENT, else a clean pass.
- [ ] **⛔ Detects a settings-level key injection (load-bearing):** an `apiKeyHelper` field present + non-empty ⇒ fault; a settings `env.ANTHROPIC_API_KEY` (or `env.ANTHROPIC_AUTH_TOKEN`) present ⇒ fault; any other member of the grounded EXTENSIBLE injecting-field set present ⇒ fault. (Exact field set + file paths grounded at Step-2.5 — see Q1.)
- [ ] **Rule 7 (§16) — presence only:** the guard checks that an injecting field EXISTS; it NEVER reads, returns, or logs the key value (an `apiKeyHelper` command is never executed; an `env.ANTHROPIC_API_KEY` value is never read). The fault code carries only the field name / file marker + a fixed code — never a value.
- [ ] **Armed-path DEGRADES, never crashes (§16/L52):** on `bootWorker`'s armed path, a settings-injection verdict routes through the SAME degrade as the env-shadowing refusal (strip the transport gate → extraction stays local/stub, ZERO cloud/subscription run) + a boot-visible code-only fault (`backends.logger.error`, rule 7). NEVER a worker-wide boot-throw.
- [ ] **Default-OFF byte-equivalent:** OFF path (`config.subscriptionArm?.enabled !== true`) does NOT read any settings file (factory-spy zero-invocation on the injected reader) — shipped default byte-identical to today's boot.
- [ ] **Fail-safe on ambiguity:** an ABSENT settings file ⇒ clean (no injection — the common case). A settings file that EXISTS but is unreadable/malformed on the armed path ⇒ DEGRADE (can't confirm clean ⇒ fail-safe, L65 ethos), code-only, never a throw.
- [ ] **Over-inclusion is fail-safe (L61/L65):** the injecting-field set is an EXPLICIT EXTENSIBLE constant; a spurious degrade only ever hits the owner-gated armed path, a missed field is a silent fail-OPEN → include when uncertain. Pin each field with a truthy-not-absent guard (L28-style).
- [ ] All unit tests pass; `/preflight` clean. No egress / spend / key read.

## RED test outline (Step 2)
Tests in `apps/worker/test/composition/subscription-settings-guard.test.ts` (pure; inject a fake settings-hierarchy reader — NO real fs/homedir):

1. **`apikeyhelper_present_faults`** (⛔ load-bearing) — settings with `apiKeyHelper: "<cmd>"` present ⇒ typed fault; the command string is NEVER executed and NOT in the fault.
   - Why: §19.5 — a settings `apiKeyHelper` injects a raw key ⇒ subscription run spends; the env guard can't see it.
2. **`settings_env_anthropic_key_faults`** — settings `env.ANTHROPIC_API_KEY` (and `env.ANTHROPIC_AUTH_TOKEN`) present ⇒ fault; value never read.
   - Why: §19.5/§7 — settings-injected env shadows the subscription (first-match-wins) invisibly to `process.env` scan at guard time.
3. **`clean_settings_pass`** — settings present with no injecting field ⇒ no fault (arm proceeds).
   - Why: non-vacuity — the guard discriminates, doesn't deny-all.
4. **`absent_settings_is_clean`** — no settings file anywhere ⇒ clean (the common default).
   - Why: absent ≠ injected; don't degrade a normal machine.
5. **`unreadable_settings_degrades_failsafe`** — a settings file that EXISTS but throws on read/parse ⇒ DEGRADE (fault), code-only, never throws.
   - Why: L65 fail-safe — can't confirm clean ⇒ deny on the owner-gated armed path.
6. **`fault_is_value_free`** (rule 7) — the fault carries only a fixed code + field/file marker, never a key value / command string.
   - Why: §16 rule 7 — never surface the credential.
7. **`each_injecting_field_individually_faults`** — parametric over `SETTINGS_INJECTION_FIELDS`; each member alone ⇒ fault (L28 per-key truthy-not-absent).
   - Why: L61/L65 — explicit-extensible set, no silent miss.
8. **`off_path_reads_no_settings`** (boot-level or guard-gating) — arm disabled ⇒ the injected settings-reader is invoked 0× (factory-spy) ⇒ byte-equivalent.
   - Why: L23/L44 — the safety gate runs ONLY on the armed path; shipped default never touches the fs.
