# /tdd brief — c5.4b_serving_context_boot_readers

## Feature
Build the two **real boot-side readers** the built `createServingContextLoader` needs to assemble a workspace's serving context from committed truth: **`CommittedVaultReader`** (reads a workspace's committed vault at head into a `CanonicalVaultSnapshot`) and **`ServingCoverageReader`** (derives the raw serving-coverage legs — parity / pin / rebuild-oracle). Both **fail-closed / never-throw**. This is the second C5.4b slice: it makes the real `admitForServing`-backed oracle **constructible from real committed truth** (Slice 3 wires it into boot behind the go-live seam, still OFF).

**HONEST-INTERIM (the C5.4a pattern):** there is **no serve-time `ParityReport` store today** (the reconciler produces reports but nothing persists/queries them at serve time), so the coverage reader's parity leg is fail-closed `undefined` ⇒ `isDegradedCoverage` ⇒ every workspace degrades in production **regardless** of the vault reader — sound + inert. Only `pinValid` is a real signal (via the built `checkVersionPin`). The reader is REAL wiring that returns real data when a parity source exists and degrades honestly when it doesn't — never a stub that fakes green. **Propose stays OFF**; nothing is armed here.

## Things to flag at Step 2.5
1. **Sync seam over async reads.** `CommittedVaultReader` is `=> CanonicalVaultSnapshot | undefined` (SYNC) but `VaultFs.read/list` + `readVaultHeadRevision` are async. Options: **(A)** change the loader seam + `readCommittedVault` to async (`=> Promise<...>` — the loader is already `async`, so this is low-friction and honest); **(B)** pre-load snapshots into a sync in-memory cache the reader closes over. My default vote: **A** — make the seam async; the loader already awaits, real fs reads are inherently async, and a per-ask sync cache adds a staleness/coherence surface. Confirm the loader-seam edit is in-scope (small, `servingContextLoader.ts` signature only — flag if it ripples).
2. **Workspace→vault mapping.** How does the reader resolve a workspaceId to its `VaultFs`/root? Options: **(A)** an injected `(workspaceId) => VaultFs | undefined` resolver (boot supplies it — single dev vault today, per-workspace roots later); **(B)** a single `vaultRoot` + `projects/<workspaceId>/` subtree convention. My default vote: **A** — an injected resolver keeps the reader agnostic to the one-repo-per-workspace vs subtree decision and matches the architecture's "one repo per workspace"; boot owns the mapping (and its WS-8 correctness) in Slice 3.
3. **`checkVersionPin` inputs at serve time.** `pin: GbrainPin` + `running: RunningGbrainVersion | undefined` + `ctx`. Reuse the same pin/running the boot startup check already resolves (Phase-11 `verifyGbrainStartup`)? My default vote: **yes, inject the same GbrainPin + a `() => RunningGbrainVersion | undefined` probe** boot already has — don't re-resolve; keeps one source of pin truth.
4. **Per-ask full-vault enumeration cost.** The reader reads the whole vault per `loadContext` call. Fine while dormant/degraded; a go-live perf concern. My default vote: **ship the simple per-ask read + a `// FUTURE-PERF` note** (caching is a go-live optimization, out of scope). Flag if you'd rather scope a cache now (I'll say no).
