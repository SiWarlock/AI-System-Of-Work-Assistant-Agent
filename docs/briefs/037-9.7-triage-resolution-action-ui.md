# /tdd brief — triage_resolution_action_ui (Phase 9, pivot slice 3)

## Feature
Add the triage-resolution **ACTION UI** to the desktop Ingestion Inbox: each parked-source card offers disposition actions that call the existing `command.disposeTriage` tRPC mutation, and on success drain the item from the inbox. Mirrors the 9.8 approval-decision flow — the renderer is UNTRUSTED (it only REQUESTS; the worker + pipeline own the replay-safe re-entry, one-writer effect, and any workspace binding). Pure local, non-HITL renderer UI + a thin renderer command-caller; no Markdown, no egress, no cross-workspace read. Code-quality review suffices (lead-confirmed).

## Acceptance criteria (what "done" means)
- [ ] A renderer command-caller (`createTriageDisposition(client)`, mirror of `createApprovalDecision`) calls `client.command.disposeTriage.mutate({ sourceId, idempotencyKey, disposition })` and returns a typed result: `{ ok: true, idempotencyKey }` on a well-formed ok, `{ ok: false }` on a typed err OR any transport throw (fail closed — a failed disposition surfaces nothing).
- [ ] The `idempotencyKey` is minted **deterministically + stably** per (sourceId, disposition) so a double-click / retry reuses the SAME key → the pipeline dedupes to one effect (the 9.7 "idempotency-key replay-safe re-entry" invariant). No fresh-per-click key.
- [ ] `IngestionInbox` renders per-card disposition action(s); activating one invokes the caller with the card's `sourceId` + the chosen disposition.
- [ ] On `{ ok: true }` the dispositioned item is removed from the inbox (drains — no re-query; `disposeTriage` returns no post-state record). On `{ ok: false }` the item REMAINS (fail closed) with a non-blocking error affordance; nothing partial/stale is shown.
- [ ] WS-8 preserved: the inbox stays workspace-scoped (empty under Global); the action acts only on a scoped item's `sourceId`. The renderer sends NO workspaceId (the command path has none — a workspace-binding disposition is the separate RecordDisposition path, out of scope).
- [ ] Render tests (jsdom tier) pin: a disposition click calls the mutation with the right args; ok → item removed; err/transport-fail → item retained; empty-state unchanged. Lib unit tests pin the fold + the deterministic key. Repo-wide `pnpm -w turbo run typecheck test` green; `/preflight` clean.
- [ ] No frozen-contract / Appendix-A change (confirm — `UiSafeIngestionItem` unchanged; `disposition` is an existing open-string command input).

## Things to flag at Step 2.5
1. **idempotency-key mint (the core design call).** The renderer must supply a STABLE key (`UiSafeIngestionItem` has none; the worker REUSES the caller's key verbatim). My default vote: **deterministic `${sourceId}:${disposition}`** — a double-click / retry reuses the key → the pipeline dedupes to one effect (directly serves the 9.7 replay-safety invariant + the `triageCommands.ts` "same key → one effect" contract). Alternatives (rejected): fresh UUID per click (NOT replay-safe); surface an `idempotencyKey` on the contract (heavier frozen-contract change; the caller-mints model is what the command expects). Confirm the key basis — if a source should be re-dispositionable to a DIFFERENT disposition after a first, keying on (sourceId, disposition) is right (different disposition → different key → new dispatch; same → idempotent).
2. **Disposition UI set (taxonomy is an arch_gap).** The command `disposition` is an OPEN string; the pipeline validates the concrete set. My default vote: **a minimal local set — "accept" / "reject"** (re-enter vs discard), defined locally as a UI convenience (like approval's 4-button union), server-re-validated. **Defer "reroute"/workspace-binding** — that is the richer RecordDisposition path (`ports/ingestionTriage.ts` `TriageDisposition` with `workspaceId`/`projectId`), which needs a workspace picker + a different command; out of this slice. Confirm the button set (flag if the pipeline expects specific disposition strings).
3. **On-ok inbox update.** `disposeTriage` returns only `{ idempotencyKey }` (no post-state UI-safe record). My default vote: **optimistic removal** of the item from the store's `ingestion` slice on ok (the item leaves the inbox; no re-query). Confirm vs a re-query of `query.ingestionInbox`.
4. **Live STREAM path — DEFERRED.** My default vote: **out of scope** (the streamed item carries no `workspaceId` to scope-filter on — the same 9.5 precedent; deferred per the carry-forward). This slice is the ACTION UI over the cold-loaded/store items only. Confirm.
5. **a11y / listbox.** My default vote: **keep the inbox a `role="list"` with per-card action `<button>`s** (standard, accessible) — do NOT convert it to a `role="listbox"` (the roving-focus a11y stays a separate future item; the inbox is not a listbox today). Confirm.
