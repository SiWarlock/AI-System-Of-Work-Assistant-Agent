# /tdd brief — durable_typed_project_registry (14.6)

## Feature
A **durable typed-Project REGISTRY**: a `@sow/db` project-registry table + dual-dialect migration + repository, a production `ResolveRegistryPort` backed by it (WS-8-scoped), and an operational project-creation path — replacing the test-only `FakeResolveRegistryPort` so `projectSync` / `deterministicProgress` / `projectDashboard` (25.3) resolve a real `ProjectIdentity` from a durable store. **Operational resolution truth ONLY** — the canonical Project (Markdown frontmatter) stays KnowledgeWriter-owned (safety rule 1); this registry is a resolution index, never a second Project writer.

- [ ] The stored row round-trips the full `ProjectRegistryEntry` shape (incl. `progressProviders[]` + `aliases[]`).
- [ ] A production `ResolveRegistryPort` (`apps/worker/src/composition/projectRegistry.ts`) over the store: `resolve(ctx)` maps `ctx.projectRef` (key OR alias) → the stored entry; unknown ref ⇒ `project_unknown`; a declared progress-provider with no `connectorId`/`remoteHandle` mapping ⇒ `provider_unmapped` (fail-closed, never a guessed source); a store fault ⇒ a typed err; **never throws** (§16).
- [ ] **WS-8 (security-reviewer=invariant):** the resolved `workspaceId` comes from the STORED entry, never a caller field; resolution is gated on the workspace being KNOWN in the 14.1 registry — a project whose workspace is ABSENT from the WS-8 registry does NOT resolve (fail-closed). `ProjectIdentity` is derived carrying NO workspaceId.
- [ ] **Rule 1 / one-writer (security-reviewer=invariant):** the creation path writes ONLY the operational registry entry — it does NOT write the canonical Project Markdown (KW-owned; the KW/copilot semantic-Project creation is deferred).
