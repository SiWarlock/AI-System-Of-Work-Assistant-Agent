# /tdd brief — serve_time_parity_report_store

## Feature
Build a durable **`ParityReportStore`** — a new operational-store repository (Drizzle `parity_reports` table + SQLite/Postgres adapters + a narrow store port + db-backed adapter) that **persists** a `ParityReport` and **queries the latest revision-scoped report** for a `(workspaceId, reconciledAtRevision)` pair. This is the missing serve-time store the C5.4b serving-coverage reader needs: today `createServingCoverageReader` returns `parity: undefined` **unconditionally** because "no serve-time ParityReport store / rebuild-oracle exists yet" (`servingContextBootReaders.ts:90-96`). This slice builds the store ONLY (write + read surface + persistence). All-local, deterministic, **NO hard line** — it stores operational truth; it does NOT wire the serving path, does NOT arm propose, does NOT touch the reader (that is the next slice).

## Acceptance criteria (what "done" means)
- [ ] New **`ParityReportRepository`** (db-layer) persists + reads a `ParityReport`:
  - `record(report, recordedAt)` inserts a row; **idempotent on `reportId`** (UNIQUE; on-conflict-no-op / first-write-wins) — a `ParityReport` is immutable operational truth, never two rows per `reportId`.
  - `getLatestForRevision(workspaceId, reconciledAtRevision)` returns the **most-recently-recorded** report for that `(workspaceId, reconciledAtRevision)`, or `undefined` when none exists. (Newest-wins so a re-reconcile at the same revision supersedes.)
- [ ] The stored row **round-trips losslessly** — a `getLatestForRevision` returns a `ParityReport` byte-equal (schema-parse-equal, incl. the embedded `divergences[]`) to what was `record`ed. Payload persisted as a JSON/text column; the frozen `ParityReportSchema.parse` gates it back on read (a corrupt/unparseable payload ⇒ fail-closed, see below).
- [ ] Repo passes ONE `repository-contract.test.ts` suite on **BOTH SQLite AND Postgres** (no dialect-specific SQL; Postgres is never a permanent stub — forbidden pattern §4).
- [ ] An **additive migration** (next number after `0005`, both dialects) creates the table under the existing backup + app↔schema-compat rule; no destructive change.
- [ ] **Fail-closed both directions (§16):** a real `DbError` on `record` OR `getLatestForRevision` is returned as a typed `Result` err — **never masked** as a silent success or as `undefined` (an absence must be a true "no row", distinguishable from a fault, so the reader can degrade on a fault without treating it as "no report"). A malformed/unparseable stored payload on read ⇒ typed err (never a half-parsed report).
- [ ] A narrow **`ParityReportStore` read-port** (`getLatestForRevision`) + a **db-backed adapter** over the repo — so the next slice's coverage reader consumes a fakeable seam, not the raw repo. (Adapter included here, mirroring brief 043 decision #4; the next slice only *binds* it.)
- [ ] All new unit + contract tests pass; repo-wide `pnpm -w turbo run typecheck test` green (+ the `SOW_PG_DOCKER` Postgres contract leg if the harness needs it); `/preflight` clean.
- [ ] **MANDATORY Step-8 review = security-reviewer + code-quality-reviewer** — the store is the serve-time source the trust oracle's coverage kill-switch reads; a false "clean report present" is a trust-gate defeat, so it is safety-relevant.
- [ ] **NO hard line crossed** — local operational-store persistence only; NOT the propose bridge, NOT the serving-reader wiring, NOT arming, no external I/O, no gbrain call.

## RED test outline (Step 2)
1. **`parity_repo_record_and_get_latest_for_revision`** — record a `ParityReport`; `getLatestForRevision(ws, rev)` returns it (schema-parse-equal, incl. `divergences[]`); an unknown `(ws, rev)` ⇒ `undefined`. *Why:* the serve-time read contract.
2. **`parity_repo_record_idempotent_on_report_id`** — a second `record` of the SAME `reportId` does not create a second row (on-conflict-no-op); exactly one row per `reportId`. *Why:* a report is immutable operational truth (§16).
3. **`parity_repo_newest_wins_same_revision`** — two DISTINCT `reportId`s reconciled at the SAME `(ws, rev)` recorded in order → `getLatestForRevision` returns the newer (by `recorded_at`). *Why:* a re-reconcile at a revision supersedes; the serve-time query must see the freshest.
4. **`parity_repo_contract_both_dialects`** — the repo passes the shared contract cases on SQLite AND Postgres (parametrized like the existing repos). *Why:* no dialect-specific SQL (forbidden pattern / §4).
5. **`parity_repo_fault_is_typed_err_not_absence`** — a forced `DbError` on read/write surfaces a typed err `Result`, NOT `undefined` and NOT a masked ok; a malformed stored payload on read ⇒ typed err. *Why:* fail-closed both directions (§16) — the reader must distinguish a fault (degrade) from a true no-row.
6. **`parity_store_adapter_satisfies_port + lossless_roundtrip`** — the db-backed `ParityReportStore` adapter satisfies the narrow port and returns a losslessly round-tripped report. *Why:* the next slice binds this fakeable seam.

## Things to flag at Step 2.5
1. **Store-port + adapter location.** Default: the `ParityReportStore` port lives where its consumer (the apps/worker coverage reader) can import it, and the db-backed adapter in `apps/worker/src/composition/…` (mirrors brief 043 — the db-backed adapter sits in the composition root). Note: `apps/worker → packages/db` IS an allowed import (unlike `packages/knowledge → packages/db`), so the adapter may live in apps/worker directly. Confirm.
2. **"Latest" ordering key.** Default: a store-side `recorded_at` ISO column (injected clock at `record`), ordered DESC for `getLatestForRevision`. `ParityReport` carries no timestamp, so ordering needs store metadata. Alternative: an autoincrement rowid. Confirm `recorded_at` (deterministic via injected clock in tests).
3. **Payload column serialization.** Default: a JSON/text column holding `JSON.stringify(report)`, re-gated through `ParityReportSchema.parse` on read (lossless round-trip tested; a parse-fail ⇒ typed err, never a half-report). Mirror existing JSON columns. Confirm.
4. **Read-port surface width.** Default: the narrow read-port exposes ONLY `getLatestForRevision` (the serve-time read); `record` stays on the repo (used by B3's write path, not the serve-time reader). Keeps the reader's seam minimal + fakeable. Confirm (vs a wider port).
5. **Idempotency semantics.** Default: UNIQUE `report_id` + first-write-wins (on-conflict-no-op), mirroring `WriteReceiptRepository`. A report id is minted per pass, so `record` is normally novel — the constraint is a defensive backstop. Confirm.
