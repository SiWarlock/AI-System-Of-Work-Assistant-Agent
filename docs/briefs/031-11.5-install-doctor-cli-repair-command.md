# /tdd brief — install_doctor_cli_repair_command (Phase 11, slice 11.5-d)

## Feature
The install-doctor **production entry / repair command** — the reachability-waiver holder that CLOSES 11.5. A composition-root that resolves the `InstallDoctorInput` config, assembles the REAL local adapters (`createLocalCommandRunner` + `createLoopbackBindProbe`), calls the built `runPrerequisiteDoctor` over the full 10-field snapshot, RENDERS the typed `DoctorReport` (per-check status · distinct repair · bounded detail · worst-of roll-up) to an injected write sink, and returns a status-derived exit code. **REPORT-ONLY — no auto-mutation, no side effects beyond the read probes + stdout** (the "repair" is printed guidance; re-running after an external fix reports that check green — idempotent by pure re-probe). LOCAL-ONLY. Deterministic (the render + exit-code core is pure over an injected report; the real adapters run behind the entry, gated in tests).

## Acceptance criteria (what "done" means)
- [ ] A PURE `renderDoctorReport(report): string` (or `string[]`) renders each check in the fixed order — `check` id · `status` · and on any non-`ok` the `failureVariant` + `repair` + (if present) `detail` — plus a final worst-of `overall` summary line. Deterministic (same report ⇒ same output).
- [ ] A PURE `doctorExitCode(overall): number` maps the roll-up to a process exit code: `ok`→0, `degraded`→0 (a TOLERATED first-class state — the app still runs), `finding`→non-zero (a prerequisite to fix). A `finding` NEVER exits 0 (an install script must be able to gate on it).
- [ ] The production entry `runInstallDoctor(deps)` (over injected `{ run, bindLoopback, config, write, now? }`, or the real adapters when unset) resolves the `InstallDoctorInput` (vaultRoot from `AppConfig`/env · `workerPrincipal` resolved in the ENTRY via `os.userInfo().username` and INJECTED — never `process.getuid()` inside the pure collector · `canonicalBrainPath`/`loopbackPorts`/`repoDir`/`localBackupAccepted` from config/documented defaults), calls `runPrerequisiteDoctor`, writes `renderDoctorReport` to the sink, and returns `doctorExitCode(overall)`.
- [ ] **REPORT-ONLY / idempotent:** the command performs NO mutation and NO side effect beyond the read probes + the write sink; re-running after a prerequisite is fixed externally reports that check green (the doctor is a pure re-probe — never auto-applies a repair).
- [ ] **Render-path redaction (rule 7):** the rendered output carries ONLY the typed `DoctorReport` fields — never a raw `ps`/`ls`/`mount` line, argv, absolute path, or secret. (`detail` is already bounded single-line ≤256 + posture details name only closed op labels; the render preserves that — no raw context added.)
- [ ] **Reachable** from the production entry — `runInstallDoctor` is the real consumer of `runPrerequisiteDoctor` → the collectors → `runDoctor`. This CLOSES the 11.5-a/b/c reachability waiver; `/wired install_doctor` traces the path from the entry.
- [ ] Wired as a runnable entry (a `bin` module + a `package.json` bin, OR a documented `pnpm` script — Step-2.5 #3) that reads env + writes stdout + sets the exit code.
- [ ] Fast-unit (inject fake `DoctorReport`s: all-ok / degraded / finding / mixed — assert rendered lines + exit code; assert the render is redaction-clean on a posture-finding fixture) + an OPTIONAL `SOW_DOCTOR_REAL=1`-gated end-to-end run (real adapters on this machine — a fail-closed finding on a dev vault is EXPECTED, asserted well-formed not green). Default suite never shells out. Repo-wide `turbo typecheck test` green; `/preflight` clean.

## RED test outline (Step 2)
Tests to write in `apps/worker/test/install/doctor-cli.test.ts`:

1. **`render_all_ok_report`** — Asserts: an all-`ok` report renders each check id + `ok`, no repair lines, and an `overall: ok` summary. Tag `spec(§13)`.
2. **`render_finding_shows_distinct_repair`** — Asserts: a report with ≥2 distinct findings renders each check's `failureVariant` + its DISTINCT `repair` (no catch-all) + `detail` when present. Why: §13 distinct-repair-per-variant.
3. **`render_redaction_clean_on_posture_finding`** — Asserts: a `stray_gbrain_process` / `vault_acl` finding fixture (whose `detail` names only closed op labels) renders with NO raw argv / path / secret substring; the output contains only the typed fields. Why: safety rule 7 render-path redaction.
4. **`exit_code_ok_degraded_zero_finding_nonzero`** — Asserts: `doctorExitCode('ok')===0`, `('degraded')===0`, `('finding')!==0`. Why: §13 install-script gating (a finding must be detectable).
5. **`run_install_doctor_composes_and_writes`** — Asserts: `runInstallDoctor` over injected fake `run`/`bindLoopback`/`config`/`write` calls `runPrerequisiteDoctor`, writes the rendered report to the sink, and returns the derived exit code (drive a mixed snapshot → a `finding` exit). Why: §13 composition + the reachability entry.
6. **`report_only_no_mutation`** — Asserts: `runInstallDoctor` performs no write/exec beyond the injected read probes + the write sink (the injected `run` sees only the fixed read commands; no repair/mutation command is issued). Why: report-only / idempotent (no auto-repair side effect).
7. **`workerPrincipal_injected_not_probed`** — Asserts: `runInstallDoctor` resolves `workerPrincipal` in the entry and passes it as config to the collector (the pure collector never calls `process.getuid`/`os.userInfo`). Why: no-inference / pure-collector (11.5-c decision 6).
8. **(gated, optional)** `real_doctor_end_to_end` — Asserts (`SOW_DOCTOR_REAL=1`): `runInstallDoctor` with the real adapters yields a well-formed rendered report + a valid exit code on this machine (a dev-vault `finding` is EXPECTED and asserted well-formed, not green). Why: the entry + real adapters genuinely exercised end-to-end.

## Things to flag at Step 2.5
1. **Exit-code semantics.** My default vote: **`ok`→0, `degraded`→0 (tolerated first-class — the app runs), `finding`→non-zero** — a finding must be gateable by an install script; degraded is not a hard failure (a §16 tolerated first-class state). Confirm (esp. whether degraded should warn-but-pass, my default, vs fail).
2. **Config resolution — reuse vs extend.** My default vote: **reuse `AppConfig`/`loadConfig` for `vaultRoot`; resolve `workerPrincipal` in the entry via `os.userInfo().username` (injected — keeps the pure collector config-driven, 11.5-c decision 6); take `canonicalBrainPath`/`loopbackPorts`/`repoDir`/`localBackupAccepted` from config or documented defaults; DO NOT expand the frozen-ish `AppConfig` unless a field is genuinely absent** (then it's a non-seam local add — flag it). Confirm the documented defaults (e.g. the gbrain pin brain path; the worker-API/Temporal/gbrain loopback ports).
3. **Entry placement — dedicated bin vs worker subcommand.** My default vote: **a dedicated `bin` (`install/bin/doctor.ts` + a `package.json` bin)** — the doctor runs as a PREREQUISITE check BEFORE the app boots, so a standalone bin is cleaner than a subcommand of the running worker (11.6 packaging + 11.7 clean-install acceptance both invoke it pre-boot). Confirm.
4. **Report-only vs auto-repair.** My default vote: **REPORT-ONLY — the doctor prints the typed `repair` guidance per finding; it NEVER auto-applies a fix** (the plan's "re-running after a prerequisite is fixed reports green WITHOUT side effects" ⇒ the fix is external; auto-mutating ACLs/mounts/processes would be a large unsafe surface). This is the safe + spec-faithful reading. Confirm (a genuine auto-repair mode would be a separate owner-gated slice).
5. **Output format — human vs `--json`.** My default vote: **human-readable lines by default + an optional `--json` mode** (the 11.7 clean-install acceptance will want a machine-parseable report). If `--json` bloats the slice, ship human-only now and defer `--json` to the 11.7 acceptance slice — flag which.
6. **Review posture.** This slice activates the safety-posture SURFACING + touches the rule-7 render path → my default vote: **dual reviewer (code-quality every-slice + security for the render-redaction / local-only / config→argv no-injection / exit-code-doesn't-mask-a-finding).** Confirm you'll invoke the security-reviewer.
