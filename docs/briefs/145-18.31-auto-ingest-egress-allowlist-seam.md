# /tdd brief — auto_ingest_egress_allowlist_seam

## Feature
Thread an **optional egress-processor allowlist** through the auto-ingest gate (`gateAutoIngest` → `AutoIngestGateOpts`/`AutoIngestWiring` → `buildAutoIngestProofSpineParams`) so the proof-spine `EgressPolicy` can allowlist the cloud processor `claude-agent-sdk`. Default-empty ⇒ byte-equivalent to today. This is the **worker seam** the desktop arming leg (18.32) forwards into; without it, an armed subscription cloud route is denied `PROCESSOR_NOT_ALLOWED`.

## Use case + traceability
- **Task ID:** 18.31
- **Architecture sections it implements:** `ARCHITECTURE.md §5` (egress veto — fail-closed processor allowlist), `§19.5` (real model transport / auto-ingest arming), `§9` (source ingestion proof-spine)
- **Related context:** Session 098 (CP3 maiden-run precondition note) — the armed `source.process` cloud `{runtime}` route (`CLOUD_EXTRACTION_ROUTE` = `{runtime:"claude-agent-sdk", egressClass:"cloud"}`) requires the workspace egress policy to allowlist `claude-agent-sdk` in **both** `allowedProcessors` AND `rawContentAllowedProcessors` (source ingestion carries raw content). Today `buildAutoIngestProofSpineParams` (`apps/worker/src/boot.ts:1101-1106`) hardcodes both lists EMPTY — there is no seam. This is an **independent OFF-lock** from `subscriptionArm`/`providerTransport` (Lessons 8/27/52): arming the subscription path without this allowlist still fails closed. Owner chose Path β (build the desktop-arming capability dormant); this is its worker foundation.

- [ ] `buildAutoIngestProofSpineParams` populates the proof-spine `resolved.egressPolicy.allowedProcessors` **and** `rawContentAllowedProcessors` from the param (source ingestion carries raw content ⇒ both lists get it), defaulting to `[]` when the param is absent/empty.

- [ ] **Assembled-broker egress pin (safety):** at the REAL assembled root, an armed personal-ws `source.process` cloud `{runtime}` route is **ALLOWED past the egress veto only when** `claude-agent-sdk` is in the allowlist; with the allowlist empty it **DENIES** `PROCESSOR_NOT_ALLOWED`. (Extend `apps/worker/test/composition/egress-veto-assembled.test.ts`.) - [ ] All unit tests pass; `/preflight` (worker `tsc --noEmit` + vitest) clean.

## Wiring / entry point (Step 7.5)
The seam is consumed on the **armed auto-ingest path**: `bootWorker` → `gateAutoIngest(opts, vaultRoot, buildAutoIngestProofSpineParams)` → `buildAutoIngestProofSpineParams` builds `resolved.egressPolicy`, which the broker's `vetoJobEgress` (rule 5, `packages/policy/src/egress.ts:138-153`) reads. **The production caller that PASSES a non-empty allowlist is the desktop forward in 18.32** — so 18.31's new param is reachability-proven via the **assembled-broker egress test** (drives the real `assembleBackends(...).broker.runJob` with the allowlist populated), not a desktop caller. State this in Step 7.5: the param's real-config caller lands in 18.32; the assembled-broker test is the reachability evidence here.

   - Asserts: both lists === `["claude-agent-sdk"]`.

3. **`assembled_broker_allows_claude_agent_sdk_cloud_route_only_when_allowlisted`** — at the real assembled root, an armed personal-ws `source.process` cloud `{runtime}` route (processor `claude-agent-sdk`): allowlist populated ⇒ passes the veto; allowlist empty ⇒ `egress_veto`/`PROCESSOR_NOT_ALLOWED`.
   - Asserts: ALLOW vs DENY discriminated on the allowlist alone.
   - Why: §5 rule 5 — the allowlist is the operative gate for a personal-ws cloud route (L50 assembled-root pin).
4. **`allowlisting_a_processor_never_bypasses_employer_raw_veto`** — employer-raw + ack-OFF + cloud route STILL denies `EMPLOYER_RAW_EGRESS_UNACKNOWLEDGED` even with `claude-agent-sdk` allowlisted.
   - Asserts: DENY unaffected by the allowlist (the employer veto precedes the allowlist).
   - Why: §5 — allowlisting is additive within rule-3; the employer-raw veto (rule at `egress.ts:119-134`) is independent (preserve the 18.30 pin).

## Things to flag at Step 2.5
1. **One param populating BOTH lists, or two separate params?** The auto-ingest path always carries raw content, so the cloud processor must be in both. My default vote: **ONE `egressAllowedProcessors` populates both lists** — separating adds config surface with no current consumer. If a future non-raw path needs a narrower raw-content allowlist, split then.
2. **Validate processor ids against a known set, or pass through as `ProcessorId[]`?** My default vote: **pass through** (branded `ProcessorId`) — the egress veto already enforces membership at the gate; a default-empty owner-set list needs no second validator. Add a defensive known-set check only if the reviewer wants belt-and-suspenders.
3. **Type the opt as `ProcessorId[]` or `string[]`?** My default vote: **`readonly ProcessorId[]`** on the worker-internal opt (type-safe), accepting the desktop leg maps its IPC `string[]` → `ProcessorId` at the worker-host boundary (18.32). Confirm the brand import doesn't create a layer issue.
