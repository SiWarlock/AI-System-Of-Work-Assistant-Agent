# /tdd brief — osb_anti_corruption_write_path_guard (task 13.1 gate `a` — the OSB anti-corruption layer: a deterministic write-path conformance guard proving NO source-extractor / vendored path can reach a vault write, + `config/osb.pin`; dormant, governance-boundary)

## Feature
Build the **task-13.1 anti-corruption layer** — the deterministic, non-HITL core that structurally enforces the OSB inheritance's one governing rule: *source extractors emit CANDIDATE DATA and NEVER write* (safety rule 1, one-writer / KN-4/KN-9). Two coupled artifacts:

1. **`packages/evals/src/osb/anti-corruption-guard.ts`** — a PURE classifier `scanForWriteSurfaces(files) → { violations, scannedCount }` that statically scans OSB source-extractor (and future `vendor/osb/**`) source text for **forbidden write-surface tokens** (any `@sow/knowledge` sole-writer import, `node:fs` write ops, `createFsVault`, atomic-vault-commit, Tool-Gateway external-write). A clean emit-only adapter has ZERO violations; a scan set that is **empty is itself a failure** (never vacuously green — the load-bearing property).
2. **`config/osb.pin`** — mirrors `config/gbrain.pin`: a parseable `key = value` pin recording the upstream OSB release tag (`v0.11.1`) + a content-SHA of the vendored subtree, so a version bump is a **deliberate act, never silent** (the artifact has zero git tags — pin release + subtree hash). Plus a PURE `parseOsbPin(text) → OsbPin` reader/validator.

This is gate **(a)** of task 13.1's 3-part eval-gate (write-path conformance). Gates **(b)** egress-leakage eval and **(c)** retrieval eval are DEFERRED (they need real `ModelProviderPort` / retrieval I/O — out of direction-C scope). Ships **DORMANT**: no real vendoring yet (no `vendor/osb/` tree), so the guard's live surface today is the shipped emit-only `*-source.ts` adapters; the pin's `subtree_sha` is an honest `PENDING_NO_SUBTREE` sentinel until real vault-agnostic libs are vendored.

## Acceptance criteria (what "done" means)
- [ ] **Forbidden-surface detection (guard core):** `scanForWriteSurfaces` flags a file whose content contains ANY write-surface token — an `@sow/knowledge` import, a `node:fs`/`fs/promises` write op (`writeFile`/`appendFile`/`mkdir`/`rename`/`rm`/`.write(`), `createFsVault`, an atomic-vault-commit reference, or a Tool-Gateway external-write (`ExternalWriteEnvelope`/`tools/adapters`) — returning the offending `{ path, token, line }`. Deterministic; no I/O of its own.
- [ ] **Clean emit-only passes:** a file that imports/uses only `registerSource`/`RegisterSourceInput`/`payloadHash`/`@sow/contracts` types + an injected transport ⇒ ZERO violations (the sanctioned tokens are NOT false-positives).
- [ ] **Never vacuously green (load-bearing):** an EMPTY scan set (or `scannedCount === 0`) is a FAILURE, not a pass — the live test asserts `scannedCount > 0` so a mis-globbed / deleted scan surface can never masquerade as "no violations."
- [ ] **LIVE conformance:** scanning the real `packages/integrations/src/connectors/adapters/*-source.ts` set ⇒ 0 violations AND `scannedCount === <the real number of *-source.ts files>` (a moved/renamed adapter that escapes the glob fails the count guard).
- [ ] **`config/osb.pin` exists + parses:** `parseOsbPin` over the real `config/osb.pin` ⇒ a valid `OsbPin`; `osb_tag === "v0.11.1"`, `osb_repo` non-empty, `subtree_sha` is `PENDING_NO_SUBTREE` (sentinel — no subtree vendored yet). Malformed pin text ⇒ typed err (never throws).
- [ ] **A bump is deliberate:** `validateOsbPin` rejects a pin missing a required key or with a `subtree_sha` that is neither the sentinel nor 40-hex — so a future subtree vendoring must record a real content-SHA to pass (the pin cannot silently drift).
- [ ] **Purity / never-throw:** both cores are pure (no clock/network/fs); every failure is a typed `Result` / verdict, never a throw.
- [ ] `packages/evals` `turbo typecheck test` green; repo-wide `pnpm -w turbo run typecheck test` green (additive `packages/evals` + a new `config/` file — no cross-package contract change, but run repo-wide as belt-and-suspenders).

## RED test outline (write cases first — `packages/evals/test/osb/anti-corruption.test.ts`)
1. `flags_knowledge_writer_import` — a synthetic file importing `@sow/knowledge` ⇒ one violation with the offending token + line. Why: safety rule 1 (sole writer).
2. `flags_fs_write_ops` — a synthetic file using `writeFile`/`fs.write(`/`createFsVault` ⇒ violations. Why: §6 no direct vault write.
3. `flags_tool_gateway_external_write` — a synthetic file referencing `ExternalWriteEnvelope` / `tools/adapters` ⇒ violation. Why: extractors are read/emit-only (§8), not external writers.
4. `clean_emit_only_no_violations` — a synthetic emit-only file (`registerSource`/`RegisterSourceInput`/`payloadHash`/contract-type imports only) ⇒ ZERO violations (sanctioned tokens not false-positived).
5. `empty_scan_set_is_failure` — `scanForWriteSurfaces([])` ⇒ `scannedCount === 0`, and the live assertion helper treats that as a FAIL (never vacuously green).
6. `live_extractors_are_clean` — read the real `adapters/*-source.ts` files, scan ⇒ 0 violations AND `scannedCount === <real file count>` (count-pinned so a glob miss fails).
7. `parses_valid_osb_pin` — `parseOsbPin` over a valid fixture ⇒ `OsbPin` with `osb_tag: "v0.11.1"`, sentinel `subtree_sha`. And over the REAL `config/osb.pin` ⇒ valid.
8. `rejects_malformed_or_undeclared_bump` — a pin missing a key / with a bad `subtree_sha` (not sentinel, not 40-hex) ⇒ typed err (a bump must record a real SHA). Malformed text ⇒ typed err, no throw.

## Things to flag at Step 2.5 (design questions — default votes)
1. **Guard mechanism — token/substring scan vs AST/import-graph.** Default vote: **line-based token scan over source text** (a simple, deterministic denylist with per-line reporting — the "grep-guard" the task names; robust to `vendor/osb/**` Python that has no TS AST). An AST/import analysis is more precise but couples to the TS compiler and can't cover vendored Python. Rationale: the guard is a structural boundary tripwire, not a type-checker; substring is sufficient + language-agnostic. Confirm.
2. **Scan glob = `adapters/*-source.ts` + `vendor/osb/**`.** Default vote: **`packages/integrations/src/connectors/adapters/*-source.ts` (auto-covers youtube/web/podcast/file/capture/url + any future source extractor) PLUS `vendor/osb/**` when present** (absent today ⇒ adapters are the live surface). Rationale: `*-source.ts` is the self-maintaining OSB-extractor family; connector adapters (calendar/drive/…) are Tool-Gateway WRITE-capable by design and are NOT in scope. Confirm whether to also include `capture-source.ts`/`url-source.ts` (default: yes — they read/ingest and must also never write).
3. **`subtree_sha` value with no vendoring yet.** Default vote: **`PENDING_NO_SUBTREE` sentinel** (honest — nothing is vendored under direction C; `validateOsbPin` accepts sentinel-OR-40hex, so a real vendoring must record a real content-SHA). Alternative (hash of empty tree) is misleading. Confirm.
4. **Denylist token set (the load-bearing list).** Default vote: `@sow/knowledge`, `markdown-vault`/`atomic-write`/`commitAtomically`, `writeFile`/`appendFile`/`mkdir`/`rename`/`rm`/`.write(`, `createFsVault`, `ExternalWriteEnvelope`/`tools/adapters`. Confirm the exact set at Step 0 by reading `packages/knowledge/src/index.ts` + `apps/worker/.../backends.ts createFsVault` (the real write surface) — the denylist must cover the sole-writer + fs + Tool-Gateway write paths without false-positiving `registerSource`/`payloadHash`/read-only fs (`readFile`/`readdir`).
5. **Where the pin parser + guard live (`packages/evals/src/osb/`).** Default vote: **`packages/evals/src/osb/`** (task 13.1's named home; `@sow/evals` already deps `@sow/integrations` + `@sow/knowledge`, so the live test can read their source paths). Confirm.
