# /tdd brief — c5.4b_serving_oracle_boot_assembly

## Feature
The C5.4b capstone: assemble the **real `admitForServing`-backed serving oracle in boot** — construct `loaderBacked = () => createServingGateOracle({ admitForServing, loadContext: createServingContextLoader({ readCommittedVault, readServingCoverage, secrets, signingKeyRef }) })` from the Slice-2 readers + an injected SecretsPort/signing-key seam + the existing gbrain-pin probe, and pass it to `selectServingOracleFactory` behind a **new default-unset arming flag** (`copilotServingOracleGoLive` → `goLiveArmed`). This makes the real oracle **constructible + proven-selectable** while staying **OFF by default** (the interim always-degraded oracle stays selected). **SAFETY-CRITICAL** (the go-live seam of the whole propose path). Own commit, no bundling.

**Three independent OFF-locks (defense-in-depth — the whole point):** even fully wired, the real oracle cannot admit anything trusted unless **ALL THREE** hold: (1) `copilotServingOracleGoLive === true` (arming flag, default unset), (2) `loaderBacked !== undefined` — which requires a **provisioned signing key** (SecretsPort/Keychain is HITL/unbuilt today, so absent ⇒ `loaderBacked` undefined ⇒ selector returns interim regardless of the flag), and (3) real KW-authored corpora + **green serving coverage** (the coverage reader degrades by reality today — no serve-time parity store). The owner flip of the arming flag is a HARD-LINE crossing — **do NOT arm it**; this slice ships it unset.

## Things to flag at Step 2.5
1. **`resolveVault` mapping strategy.** Single dev `createFsVault(vaultRoot)` for the served workspace(s) vs a per-workspace-root resolver (architecture's one-repo-per-workspace). My default vote: **an injected `(workspaceId) => VaultFs | undefined` resolver, fail-closed on unknown**, built in boot from the configured vault root(s); for the single-vault dev reality, map only the well-known served workspace(s) and return `undefined` for the rest (never a shared cross-workspace vault). Confirm the WS-8 stance (a workspace must never resolve to another's vault).
2. **When to construct `loaderBacked`.** Always (and let the gate fail-close on an unresolved key) vs only-when-a-signing-key-is-configured (⇒ undefined otherwise). My default vote: **only-when-configured** — a `loaderBacked` that can never verify a signature is dead weight; leaving it `undefined` makes OFF-lock 2 structural (the selector's `loaderBacked!==undefined` guard) rather than relying on a runtime withhold. Both are safe; this is cleaner.
3. **SecretsPort seam source.** Inject the knowledge-local `SecretsPort` (`provenance-stamp.ts`) via new optional `BootConfig` fields (default absent) vs wait for a canonical policy-layer SecretsPort. My default vote: **inject the knowledge-local one via `BootConfig` (default absent)** — it's the same interface the writer's stamp-mint uses; the canonical policy-layer SecretsPort + real Keychain adapter is the deferred HITL item (11.4), and this seam is where it plugs in. Note the `arch_gap` in-code.
4. **e2e green-coverage injection.** The crown-jewel test injects a GREEN coverage reader (since the real one degrades by reality) to exercise trusted admission through the boot-constructed oracle. My default vote: **yes — inject green coverage in test #5, and keep test #6 on the REAL reader to prove real-coverage-degrades-even-when-armed.** Confirm this is an honest proof (the real coverage gap is a separately-tracked go-live precondition, not hidden).
