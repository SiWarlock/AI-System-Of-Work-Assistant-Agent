## Things to flag at Step 2.5
1. **Which existing `FailureClass` does `pass_faulted` mint?** The sibling `skipped_derive_error` branch already uses `parity_defect`. A `pass_faulted` is a durable-store `record` / health-sink fault during the reconcile pass — not strictly a data-parity divergence, but same subsystem. Candidates: `parity_defect` (sibling-consistent; the `subjectRef` + `detail` code distinguish it) vs a least-wrong member + a greppable `arch_gap` marker in the message (Lesson 18 — names the absent dedicated store-fault class). `worker_down` is OUT (reserved for genuine infra, Lesson 18); `write_through_failed` is OUT (that means an external Tool-Gateway write-through, not an internal operational-store write). **My default vote: `parity_defect`** — sibling-consistent, keeps reconcile health items in one class, and the `detail` code + `subjectRef` disambiguate; add an `arch_gap`-style token in the safe message (e.g. `"pass_faulted"`/`"store_fault"`) so the store-fault cause is named without inventing an enum member.
2. **Does `summarizeOutcome` already emit a safe `detail` for `pass_faulted`?** If yes (the redacted causeCode), 7b mirrors the `skipped_derive_error` branch exactly. If `detail` is absent/optional for `pass_faulted`, use a fixed safe literal (e.g. `"pass_faulted"`) for the message code — **never** reach past the summary to the raw cause. **My default vote: reuse `summary.detail` when present, else a fixed `"pass_faulted"` literal** — both are safe-by-construction.
3. **Scope of the 7a integration test (RED #3).** If a snapshot that cleanly reconciles yet emits a routed HealthItem is hard to fixture, is the piece-A `health_sink_fault_propagates` unit test (already green) + a driver-level `pass_faulted`-on-rejecting-sink assertion sufficient coverage? **My default vote: yes** — don't contort a fixture; the A-level contract is already pinned, so a driver/scheduler-level assertion that a rejecting sink → `pass_faulted` (no escape from `flush`) is the right boot-level pin.
