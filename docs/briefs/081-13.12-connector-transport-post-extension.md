# /tdd brief — connector_transport_post_extension (connector round 6 slice 1 — additive POST+body on the template)

## Feature
Extend the shared connector HTTP transport template with an **optional POST method + JSON request body**, additively — so a GraphQL-over-POST connector (Linear, slice 2) can be built over it. The 5 existing GET connectors (Asana/Drive/Calendar/Granola/GitHub) stay **byte-unchanged** (method defaults to GET; no body, no content-type). This is the security-sensitive half of round 6 — a POST path on the transport — so it ships + dual-reviews on its own before the Linear spec layers on top.

**Why the extension.** The template's `HttpTransportRequest.method` is the literal `"GET"` and there is no body. Linear's API is GraphQL — a single `POST https://api.linear.app/graphql` with the query in the request body. The template needs to (a) carry an optional POST method + a spec-built JSON body, and (b) keep every safety property that holds for GET holding identically for POST: **SSRF guard on the FINAL url regardless of method**, token **Authorization-header-only** (never in the body), redacted faults, and the wrapped-callback boundary (a throwing body-builder can't escape unredacted).

**Read-only intent preserved.** The POST capability is mechanical; a POST connector's read-only-ness (ING-7 / rule 6) is enforced at the SPEC level by a fixed query-only body (slice 2's Linear invariant), not by the transport. The template documents this. GET stays the default so no existing connector gains a write-method path.

**Dormant + no hard line.** Real transports/secrets stay UNBOUND (byte-equivalent); binding is the owner's arming crossing.

  - Token stays Authorization-header-only (never in body/url) — rule 7 unchanged.

## RED test outline (`connector-http-transport.test.ts`)
1. **`post_spec_sends_method_body_content_type`** — a fake spec `{method:"POST", buildBody:()=>'{"query":"..."}'}` ⇒ the `HttpTransportRequest` handed to the transport has `method:"POST"`, the built `body`, and `content-type: application/json`; a 2xx maps via `mapPage`.
2. **`get_specs_byte_unchanged`** — a spec with no `method` ⇒ `method:"GET"`, NO `body` key, NO `content-type` header (the existing GET request shape, byte-identical). (The 5 vendor suites staying green is the rest of the proof.)
3. **`ssrf_guard_method_agnostic_on_post`** — a POST spec whose FINAL url host is off-allowlist ⇒ `isAllowedRemoteEndpoint` refuses it, ZERO token read, ZERO dispatch, ZERO `buildBody` call (guard fires first).
4. **`buildBody_throw_fails_closed`** — a POST spec whose `buildBody` throws ⇒ redacted `TransportFailure` (host-ref only; no raw content); `buildBody` absent while `method:"POST"` ⇒ fail-closed failure.
5. **`post_body_is_token_free`** — a POST spec + a fake `SecretsAccessor` returning a marker token ⇒ the `HttpTransportRequest.body` contains the marker token NOWHERE; the token appears ONLY in the `Authorization` header (rule 7 on the new body path).

## Things to flag at Step 2.5
1. **`buildBody` receives `TransportRequest` (token-free), same as `buildQuery`/`mapPage`** — the GraphQL variables (cursor/pageSize) come from `request.cursor` + spec constants, never the token. Confirm the seam.
2. **GET default is byte-exact** — confirm no stray `content-type`/`body` leaks onto a GET request (the shape the 5 existing connectors + their suites assert).
3. **SSRF-before-body** — confirm the guard runs before `buildBody` is called (so a throwing/expensive body-builder never runs for an off-allowlist url).
