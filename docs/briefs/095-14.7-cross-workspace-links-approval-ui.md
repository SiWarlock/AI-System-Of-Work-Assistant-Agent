# /tdd brief — cross_workspace_links_approval_ui (14.7 desktop leg)

## Feature
The **desktop cross-workspace-links surface** — the owner-facing approval UI for the SINGLE sanctioned WS-8 cross-workspace read input (safety **rule 4**). It drives the already-shipped worker procedures `crossWorkspaceLink.create` / `.approve` / `.revoke` (`06ba3d3a`): create a PENDING directional link (from → to) with a bounded scope, **owner-approve** it (the deliberate human-in-the-loop authorization that opens the scoped cross-read path), and **revoke** it (closes the path immediately). Renders ONLY the UI-safe link summary (ids + scope descriptors + status + timestamps) — never any raw cross-workspace content. **Desktop-track; ISOLATED (rule-4 surface — not bundled), security-reviewer=invariant.**

## Acceptance criteria (what "done" means)
- [ ] A new surface (`apps/desktop/renderer/surfaces/cross-workspace-links/`) that: **creates** a link (from → to + bounded scope, minting a `linkId`), lists the session's links with status, **approves** a pending link, and **revokes** a link — driving `crossWorkspaceLink.create/approve/revoke`.
- [ ] **Deliberate rule-4 authorization:** the approve affordance presents the FULL `(fromWorkspaceId → toWorkspaceId, scope)` explicitly before the owner authorizes — approve is an explicit action, never automatic, never a bulk/implicit approve. (This UI is the single human-in-the-loop gate that opens a cross-read path.)
- [ ] **Renders only UI-safe fields (rule 4/7):** the surface renders ONLY `UiSafeCrossWorkspaceLink` fields (ids, scope descriptors, status, timestamps). It renders ZERO raw cross-workspace content (there is none in the shape — pin that the surface reads only these fields and invents no content read).
- [ ] **WS-8 selectors:** the from/to workspace pickers offer only registered/onboarded workspaces; the surface never submits an arbitrary/free-form unregistered id. A self-link (from===to) is prevented client-side too (the worker also rejects `cross_workspace_link_self`).
- [ ] **No pre-approval smuggling:** `create` submits only the whitelisted create fields — never a `status`/`approvedAt` (owner approval stays a separate explicit transition; the worker strips it, the UI must not rely on that).
- [ ] **Revoke is immediate + terminal in the UI:** a revoked link shows revoked and offers no re-approve (mirrors the worker's terminal transition).
- [ ] **Safe error states (§16 / rule 7):** each typed failure (`workspace_unknown`, `cross_workspace_link_self`, `cross_workspace_link_invalid_scope`, `cross_workspace_link_immutable`, `link_not_pending`, `store_fault`/`degraded_unavailable`) surfaces a user-visible safe state — never a raw driver cause / secret.
- [ ] **Reachable:** the surface is routed + mounted in AppShell (a real production entry, not test-only).
- [ ] Deterministic tests pass (component/store tests for create/approve/revoke wiring, UI-safe-only render, WS-8 selectors, safe error states); `/preflight` clean (desktop + repo-wide).

## RED test outline (Step 2) — deterministic flow (visual layer is `/design-review`)

5. `renders_only_ui_safe_link_fields` — the surface reads/renders only `UiSafeCrossWorkspaceLink` fields; no raw-content read path exists. Why: rule 4/7 (never raw cross-workspace bytes at the renderer).
6. `workspace_selectors_only_offer_registered_workspaces` — the from/to pickers offer only onboarded/registered workspaces; self-link is prevented client-side. Why: WS-8 defense-in-depth.
7. `typed_failure_shows_safe_error_state` — each typed error (workspace_unknown / self / invalid_scope / immutable / not_pending / store_fault) surfaces a safe user state, no raw cause. Why: §16 / rule 7.
