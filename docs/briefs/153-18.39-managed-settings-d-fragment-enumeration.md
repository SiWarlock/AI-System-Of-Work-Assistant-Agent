# /tdd brief — managed_settings_d_fragment_enumeration

## Feature
Close the second GATE-1 fail-open: the armed-path settings-injection guard's real reader (`readClaudeCodeSettings`, 18.36) reads only the **4 flat settings tiers** (managed `managed-settings.json` / project-local / project / user), but the real runtime **claude-code 2.1.201** ALSO reads a **`managed-settings.d/*.json` drop-in fragment directory** in the managed tier (base then sorted `*.json`, deep-merged). So a managed-settings.d fragment injecting `apiKeyHelper` / a cred-minting field / an `env.ANTHROPIC_API_KEY` shadow is **invisible** to `assertNoSettingsKeyInjection` — a raw-API-key injection the guard can't see (real metered spend on a "subscription" run). Extend `readClaudeCodeSettings` to ALSO enumerate the `managed-settings.d/*.json` fragments as additional `managed`-tier settings sources, so the existing presence-only injection scan (`SETTINGS_INJECTION_FIELDS` + the single-sourced `SUBSCRIPTION_SHADOWING_ENV_KEYS` env-block leg) inspects them too. Presence-detection per fragment is sufficient (no deep-merge needed — any injecting fragment ⇒ fault). Fail-safe: an unreadable/malformed fragment degrades (like the flat tiers). Byte-equivalent when unarmed (the reader is invoked ONLY on the armed boot path via `guardSettingsOnArmedPath`). Presence-only, rule 7 (never the value / never executes `apiKeyHelper`). Arms NOTHING; closes session-104 residual #40 — a §ARM-18 CHECKPOINT-1 pre-first-armed-run hardening.

## Acceptance criteria (what "done" means)
- [ ] `readClaudeCodeSettings` enumerates `managed-settings.d/*.json` fragments in the managed tier (alongside the base `managed-settings.json`), each returned as a `managed`-marker `SettingsSourceRead` — so `assertNoSettingsKeyInjection` scans them via the EXISTING presence logic (no change to the scan itself).
- [ ] **A fragment injecting a cred-minting field degrades:** a `managed-settings.d/x.json` with `apiKeyHelper` (or any `SETTINGS_INJECTION_FIELDS` member) present+non-empty ⇒ `assertNoSettingsKeyInjection` returns a fault (marker `"managed"`), the armed path degrades fail-closed.
- [ ] **A fragment env-block shadow degrades (single-source):** a fragment with `env.ANTHROPIC_API_KEY` (or any `SUBSCRIPTION_SHADOWING_ENV_KEYS` member — incl. the 18.38 additions) present ⇒ fault (the env-block leg reaches fragments too).
- [ ] **Fail-safe on a bad fragment (§16/L65):** an unreadable / malformed / non-object `managed-settings.d/*.json` ⇒ `unreadable` outcome ⇒ degrade (never a throw, never a silent skip).
- [ ] **No false-degrade when clean:** an absent `managed-settings.d/` dir, an empty dir, or a dir of clean fragments ⇒ NO fault (the common case stays clean). Non-`*.json` files in the dir are IGNORED.
- [ ] **Deterministic enumeration:** fragments read in a deterministic (sorted) order (presence-verdict is order-independent, but the read order is stable for reproducibility).
- [ ] **Settings-path RELOCATION handled fail-safe (routed from 18.38's Step-8 review):** `CLAUDE_CODE_MANAGED_SETTINGS_PATH` + `CLAUDE_CODE_REMOTE_SETTINGS_PATH` RELOCATE where the runtime reads managed settings — so the guard's default-path scan can miss a relocated injecting file (fail-open). Handle it: **default = present-on-the-armed-path ⇒ DEGRADE** (fail-safe — a relocation var set means the scan can't be trusted; simplest + closes the fail-open). Whether this degrade lives in the env-guard leg (present-check) or the settings reader is the implementer's Step-2.5 call (Q5); honoring the redirect (scan the relocated file + its `.d`) is a nicer future state but NOT required for GATE-1 CLEAR. Operability caveat (deployment legitimately relocating settings can't arm) → flip-runbook reconciliation, no exposure.
- [ ] **Session-env `*.sh` HOOKS detected (routed from 18.40's Step-8 review — a 2nd managed-settings injection vector):** claude-code MANAGED settings can define **session-env / hook scripts (`*.sh`)** that RUN on session start and can set env / mint creds — and (like the managed tier) they **survive `settingSources:[]`**, so the env-scrub (18.40) does NOT cover them and a scrubbed child still runs a managed hook. The settings-injection guard must DETECT the presence of a session-env-hook / hook-command field in the settings hierarchy (PRESENCE-only ⇒ degrade — we can't scan what the `*.sh` does, so a present hook that could inject is fail-safe-degraded), same model as `apiKeyHelper`. Ground the exact field/mechanism at Step-2.5 (Q6). This is part of the settings-file surface completeness that gates GATE-1 CLEAR alongside the fragment enumeration.
- [ ] **Shipped default byte-equivalent:** arm unset ⇒ `guardSettingsOnArmedPath` short-circuits ⇒ the reader is NEVER invoked ⇒ zero fs reads of the fragment dir (factory-spy zero-invocation, L23/L44). No new read outside the armed path.
- [ ] **In-code note updated:** replace the "managed-settings.d not enumerated — a re-verify residual" note with the implemented behavior + keep the "re-verify the fragment path/merge vs the SDK-bundled version at the flip" HARD-precondition note.
- [ ] `/preflight` clean. Presence-only, rule 7 (no value read, `apiKeyHelper` never executed).

## Things to flag at Step 2.5
1. **⛔ Confirm the EXACT fragment path + read semantics vs the SDK-bundled 2.1.201 binary.** You grounded that 2.1.201 reads `managed-settings.d/` (base then `*.json` sorted, deep-merge). Confirm before GREEN: (a) the dir path — is it `/Library/Application Support/ClaudeCode/managed-settings.d/` (sibling of the base `managed-settings.json`)? (b) is it ONLY the managed tier that has a `.d` dir, or do other tiers too (project/user)? My default: managed tier only (the enterprise-policy tier is the documented `.d` case); flag if the binary shows others. (c) We only need PRESENCE per fragment (not the merged value) — confirm a fragment's field presence alone triggers the real runtime's honoring (it does, since a merged `apiKeyHelper` is honored regardless of which fragment supplied it). My default: scan each fragment independently for presence (simpler + strictly fail-safe vs a deep-merge).
2. **`readManagedFragments` fault semantics.** A fragment dir that exists but `readdirSync` throws (perms) ⇒ my default: emit ONE synthetic `unreadable` `managed` source (fail-safe degrade — can't confirm the fragments are clean), never silently skip. Confirm.
3. **Scope — managed tier only, or all tiers with a `.d`?** Default: managed only (Q1b). If the binary reads e.g. a project `.d`, INCLUDE it (over-inclusion fail-safe) — flag the finding.
4. **Deep-merge vs per-fragment presence.** Default: per-fragment presence-scan (no merge) — a merge is only needed if a fragment could REMOVE a field another sets (settings don't delete on merge for these fields), so per-fragment presence is both simpler and can't miss an injection. Confirm no removal semantics defeat it.
5. **Settings-path relocation (`CLAUDE_CODE_MANAGED_SETTINGS_PATH`/`REMOTE_SETTINGS_PATH`) — where to degrade.** Routed here from 18.38's Step-8 review (they relocate the settings file, a settings-reader concern, not an auth/egress env shadow). Decide the home: (a) a present-check in the env-guard leg (mirrors `ANTHROPIC_CONFIG_DIR` — smallest change), or (b) the settings reader degrades when a relocation var is set (semantically cleaner, keeps the env set = auth/egress). My default: **(b)** so the 18.38 env set stays auth/egress-shadows only and the settings guard owns the full settings-injection surface. Honoring the redirect (scan the relocated file) is NOT required for GATE-1 CLEAR — a noted follow-up. Flag your pick.
6. **⛔ Session-env `*.sh` HOOKS — ground the exact field + detect presence.** The 18.40 reviewer surfaced that managed settings can define session-env / hook scripts (`*.sh`) that run on session start (survive `settingSources:[]`, so env-scrub doesn't cover them).
