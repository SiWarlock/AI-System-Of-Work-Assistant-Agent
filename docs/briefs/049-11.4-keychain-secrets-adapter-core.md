# /tdd brief — keychain_secrets_adapter_core

## Feature
The foundation of the real macOS-Keychain `SecretsPort` adapter (task 11.4): a **mockable Keychain backend seam** + the pure `createKeychainSecretsAdapter` core implementing `@sow/knowledge`'s `SecretsPort.resolveSigningKey(ref) => Promise<Result<Uint8Array, SecretUnresolved>>` over the injected backend. It parses/validates an opaque `keychain://…` reference, dispatches to the backend, maps every failure mode (missing / locked / denied / backend-error) to a typed `SecretUnresolved` carrying **only the ref + a fixed class token — never the key material or raw stderr**, and never throws. **This slice does NO real Keychain I/O** — the build + all tests run against a FAKE in-memory backend (the real macOS `security`-CLI backend is Slice 2, behind this same seam). This is the highest-leverage brick: `resolveSigningKey` is the C5.4b OFF-lock-2 signing-key source, and the same backend core later serves provider API keys.

**SAFETY-CRITICAL — top-tier secrets surface (safety rule 7):** secrets resolve ONLY through `SecretsPort`; the resolved bytes are NEVER placed in a typed error, a log, or the renderer; the adapter holds them only in a local binding to return. Own commit, no bundling. MANDATORY security + code-quality review (key handling + every error/log path for leakage).

## Acceptance criteria (what "done" means)
- [ ] A `KeychainBackend` seam (injected, async, mockable) is defined — the narrow macOS-Keychain read surface, e.g. `read(service, account) => Promise<Result<Uint8Array, KeychainBackendError>>` where `KeychainBackendError.kind ∈ {not_found, locked, denied, backend_error}`. Total (never throws); shape per Step-2.5 Q2/Q3.
- [ ] `createKeychainSecretsAdapter(backend): SecretsPort` — `resolveSigningKey(ref)` parses+validates the `keychain://<service>/<account>` ref, dispatches to `backend.read`, and returns `ok(Uint8Array)` on a hit.
- [ ] Every failure maps to a typed `SecretUnresolved` (never a throw): a malformed/invalid ref ⇒ err WITHOUT a backend call (fail-closed); backend `not_found`⇒reason `"missing"`, `locked`⇒`"locked"`, `denied`⇒`"denied"`, `backend_error`⇒`"backend_error"`. `reason` is a FIXED class token only.
- [ ] **Rule-7 leakage pins (the load-bearing safety):** the resolved key bytes NEVER appear in the returned `SecretUnresolved` (it carries only `ref` + the class token), and the adapter passes NO secret material to any log/sink. A backend error's raw stderr/detail is DROPPED (not copied into `reason`). Assert a returned err over a fixture ref contains neither the fixture key bytes nor any backend raw-detail string.
- [ ] The adapter NEVER throws across the boundary — a backend that itself throws (contract violation) is caught → `SecretUnresolved{reason:"backend_error"}`.
- [ ] Ref parsing is charset/format fail-closed (a ref with traversal/injection/empty segments ⇒ invalid ⇒ err, no backend call) — mirrors the WS-8 segment-guard discipline.
- [ ] All unit tests in `apps/worker/test/secrets/keychain-adapter.test.ts` (NEW) pass over a FAKE in-memory backend; `/preflight` clean; repo-wide `pnpm -w turbo run typecheck test` green (no cross-package consumer yet — the adapter is wired in the boot slice).
- [ ] MANDATORY security + code-quality review CLEAR (secrets surface: key handling, error/log leakage, ref-injection).

## RED test outline (Step 2) — `apps/worker/test/secrets/keychain-adapter.test.ts` (fake in-memory backend)
1. **`resolves_signing_key_bytes_on_backend_hit`** — a fake backend returns bytes for `(service,account)`; `resolveSigningKey("keychain://sow/kw-signing")` ⇒ `ok(Uint8Array)` equal to the fixture bytes. Why: §16 resolve-by-reference.
2. **`missing_locked_denied_map_to_typed_reason`** — backend `not_found`/`locked`/`denied` ⇒ `err SecretUnresolved` with `reason` `"missing"`/`"locked"`/`"denied"` respectively + `ref` echoed. Why: §16 typed-unavailable, LIFE-6 degraded inputs.
3. **`malformed_ref_fails_closed_without_backend_call`** — refs like `""`, `not-a-keychain-ref`, `keychain://`, `keychain://a/../b`, `keychain://a/b/c` ⇒ err, and the fake backend's `read` was NEVER invoked (spy asserts 0 calls). Why: fail-closed + ref-injection defense.
4. **`err_never_carries_key_bytes_or_raw_backend_detail`** (the safety pin) — a backend returning `backend_error` WITH a raw-detail string that contains a secret-shaped substring ⇒ the returned `SecretUnresolved` contains only `{code, ref, reason:"backend_error"}` — assert it does NOT contain the fixture key bytes NOR the raw-detail string. Why: safety rule 7 (no secret/raw material in a typed error).
5. **`never_throws_on_backend_throw`** — a backend whose `read` REJECTS/THROWS ⇒ caught ⇒ `err SecretUnresolved{reason:"backend_error"}`, no throw across the boundary. Why: §16 no-throw.
6. **`satisfies_the_knowledge_SecretsPort_contract`** — the returned object is assignable to `SecretsPort` (structural), so it drops into `StamperDeps.secrets` / the `provenanceServingOracle` bundle. Why: it must be the real OFF-lock-2 signing-key source.

## Things to flag at Step 2.5
1. **One adapter, one or two facades?** `resolveSigningKey` (bytes, `SecretsPort`) is this slice's focus; the providers' `getSecret` (string, `SecretsAccessor`) is the sibling API-key facade over the SAME backend read. My default vote: **build the backend-read core + the `resolveSigningKey` facade now; add the `getSecret` facade in the boot slice** (where providers wire in) — keeps this slice focused + safety-review tight on the signing-key path. Flag if you'd rather expose both facades here (cheap, same core) — acceptable if it doesn't widen the review surface.
2. **Backend seam shape — `read(service, account)` vs `read(ref)`.** My default vote: **the adapter parses the ref; the backend takes structured `(service, account)`** — keeps the backend dumb (a thin `security`-CLI wrapper in Slice 2) and the ref-parsing/injection-guard unit-testable in the adapter (where the fail-closed pins live).
3. **`SecretUnresolved.reason` vocabulary.** A fixed closed set (`missing`/`locked`/`denied`/`backend_error`) vs an open string. My default vote: **a fixed class token** — redaction-safe by construction (an open reason risks a caller/backend leaking detail into it), and it maps cleanly onto the Keychain-locked controller's inputs in the boot slice.
4. **Bytes handling.** `resolveSigningKey` returns `Uint8Array` (raw HMAC key). Confirm the adapter does not stringify the bytes (no `.toString()` that could hit a log) and holds them only transiently. My default vote: **pass the backend's `Uint8Array` straight through; never stringify.** Reviewer confirms no intermediate string copy of key bytes exists.
