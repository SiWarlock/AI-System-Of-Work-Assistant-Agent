# /tdd brief — keychain_security_cli_backend

## Feature
The real macOS-Keychain read backend behind the Slice-1 `KeychainBackend` seam: `createSecurityCliKeychainBackend({ exec })` calls the macOS `security find-generic-password` CLI (via an **injected, mockable execFile-shaped `exec` seam** — args-array, NEVER a shell string) and maps its exit code / stderr to the typed `KeychainBackendError{kind, detail?}`. **This slice does NO real Keychain I/O in build or test** — every test drives a FAKE `exec` that returns synthetic `{code, stdout, stderr}`; the real `security` binary runs only when the owner provisions + wires it (Slice 3 + owner-gated). **SAFETY-CRITICAL (rule 7):** the CLI's `-w` stdout IS the secret value — it is returned as `Uint8Array` and NEVER logged; `KeychainBackendError.detail` carries only a bounded, secret-free stderr/code summary (and the Slice-1 adapter drops it entirely). Own commit, no bundling. MANDATORY security + code-quality review.

## Acceptance criteria (what "done" means)
- [ ] `createSecurityCliKeychainBackend({ exec }): KeychainBackend` where `exec` is an injected `(file: string, args: readonly string[]) => Promise<{ code: number; stdout: Uint8Array | string; stderr: string }>` (execFile-shaped; NO shell). The backend invokes `security` with an **args array** (e.g. `["find-generic-password", "-w", "-s", service, "-a", account]`) — never a concatenated shell string.
- [ ] **Success:** `exit 0` ⇒ `ok(Uint8Array)` = the `-w` stdout value with exactly the CLI's single trailing newline stripped (nothing else trimmed — a key may legitimately contain other whitespace bytes). Encoding contract per Step-2.5 Q3.
- [ ] **Fault mapping (typed, never throws):** `exit 44` (errSecItemNotFound) ⇒ `not_found`; a locked signal (errSecInteractionNotAllowed / "interaction not allowed" / the documented locked code) ⇒ `locked`; an auth-denied signal ⇒ `denied`; anything else (non-zero, unrecognized) ⇒ `backend_error`. The exact locked/denied codes are macOS-version-specific → see Step-2.5 Q2 (robust-by-stderr-pattern + a live-verify note).
- [ ] **Rule-7 leakage pins (load-bearing):** the `-w` stdout (the secret) is NEVER placed in `detail`, a log, or anywhere but the `ok` `Uint8Array`; `detail` on a fault is a BOUNDED (≤N chars), secret-scrubbed summary of stderr + the exit code — assert a fault whose (synthetic) stderr contains a secret-shaped substring yields a `detail` that does NOT contain it, and that a SUCCESS path never logs stdout.
- [ ] **Never throws:** an `exec` that rejects/throws (spawn failure, ENOENT for `security`, timeout) ⇒ `backend_error` (caught) — no throw across the seam.
- [ ] All unit tests in `apps/worker/test/secrets/keychain-backend.test.ts` (NEW) pass over a FAKE `exec`; `/preflight` clean; repo-wide `pnpm -w turbo run typecheck test` green.
- [ ] MANDATORY security + code-quality review CLEAR (CLI-boundary secret handling; argv-injection; no shell; stdout never logged).

## RED test outline (Step 2) — `apps/worker/test/secrets/keychain-backend.test.ts` (fake `exec`)
1. **`success_returns_value_bytes_minus_trailing_newline`** — fake `exec` returns `{code:0, stdout:"<key>\n"}`; backend ⇒ `ok(Uint8Array("<key>"))` (exactly one trailing `\n` stripped; interior bytes untouched). Why: §16 resolve-by-reference; encoding fidelity.
2. **`invokes_security_with_an_args_array_no_shell`** — assert the fake `exec` was called with `("security" | the resolved bin, ["find-generic-password","-w","-s",service,"-a",account])` — an ARRAY, never a shell string; service/account passed as discrete argv. Why: argv-injection defense (no shell).
3. **`exit_44_maps_not_found`** — `{code:44}` ⇒ `err {kind:"not_found"}`. Why: §16 typed unavailable.
4. **`locked_and_denied_signals_map_typed`** — a locked signal ⇒ `{kind:"locked"}`; an auth-denied signal ⇒ `{kind:"denied"}` (per the Q2 mapping). Why: LIFE-6 degraded inputs.
5. **`unrecognized_nonzero_maps_backend_error`** — an unknown non-zero code ⇒ `{kind:"backend_error"}`. Why: fail-safe default.
6. **`detail_is_bounded_and_secret_scrubbed`** (the safety pin) — a fault whose synthetic `stderr` contains a secret-shaped substring (and the fixture "value") ⇒ the returned `detail` is ≤N chars and contains NEITHER the secret-shaped substring NOR the value; and NO code path passes stdout to `detail`. Why: rule 7.
7. **`exec_throw_or_reject_maps_backend_error`** — `exec` rejects (ENOENT/timeout) ⇒ `{kind:"backend_error"}`, no throw. Why: §16 no-throw.
8. **`success_never_stringifies_or_logs_the_value`** — with an injected log spy (if the backend takes one) OR by construction, assert no `.toString()`/log of the `-w` stdout on the success path. Why: rule 7 (the value never hits a sink).

## Things to flag at Step 2.5
1. **`exec` seam shape.** An injected `(file, args) => Promise<{code, stdout, stderr}>` (execFile-shaped, mockable) vs importing `node:child_process` directly. My default vote: **inject the execFile-shaped seam** — no real spawn in test, the parse is unit-testable, and Slice 3 supplies a thin real `execFile` wrapper (bounded timeout + maxBuffer). Confirm stdout typing (`Uint8Array` preferred so a binary key round-trips without a lossy string step).
2. **Exit-code → kind mapping robustness.** The exact macOS codes: `44` = errSecItemNotFound (well-known); locked = errSecInteractionNotAllowed (`-25308`, but the CLI's *exit* code differs by macOS version); denied = errSecAuthFailed. My default vote: **map `0`→ok, `44`→not_found, then classify locked/denied by a STDERR PATTERN (case-insensitive "interaction not allowed"/"locked" ⇒ locked; "auth"/"denied"/"not authorized" ⇒ denied), else `backend_error`** — robust across versions; add an in-code note that the exact numeric codes must be verified against the live `security` binary at owner-provisioning (the mocked tests pin the classifier, not the real codes). Flag if you'd rather hard-code numeric codes.
3. **Key-encoding contract (`-w` output).** `security -w` prints the generic-password data + a trailing newline. A raw-binary HMAC key may not survive as a clean `-w` string. My default vote: **strip exactly ONE trailing `\n` and return the remaining stdout bytes as `Uint8Array`; document that the owner stores the signing key in a `security`-safe encoding (e.g. base64) and the DECODE (if any) is a boot-slice/go-live concern** — this slice returns the raw `-w` bytes minus the newline; the encoding/decode contract is pinned when the owner provisions. Flag if you want the base64-decode here (I lean: no — keep the backend a thin reader; decode at wiring).
4. **`detail` scrubbing.** What goes in `KeychainBackendError.detail`? My default vote: **the exit code + a bounded (≤200-char) stderr snippet with any secret-shaped run stripped** (the Slice-1 adapter drops `detail` anyway, so it's only for backend-isolation debugging — keep it minimal + safe). Never stdout.
