# /tdd brief — native_dotenv_allowlisted_loading

## Feature
Load a repo-root `.env` natively in Electron main at startup, hydrating **only the recognized `SOW_*` config allowlist** into `process.env` (never blanket) — so the app reads config directly and the `dev.sh` wrapper's blanket `source .env` becomes redundant. The allowlist **structurally excludes** the subscription-shadowing env set + all real secrets (none are `SOW_*`), so a plaintext `.env` can never shadow the Claude subscription, redirect egress, or auto-load a secret. Default-safe: missing `.env` ⇒ no-op; existing `process.env` wins.

## Use case + traceability

- **Related context:** Today `dev.sh` (repo root) does `set -a; source .env; set +a` — a **blanket** source that would hydrate ANY key incl. `ANTHROPIC_API_KEY`/proxies/secrets (the shadow + egress-redirect risk). `apps/desktop/main/index.ts:61-90` reads `process.env["SOW_*"]` and threads to the worker over IPC. The 13-var shadowing set is `SUBSCRIPTION_SHADOWING_ENV_KEYS` (`apps/worker/src/composition/subscription-auth-guard.ts:56`, exported) — the armed-boot guard (`assertSubscriptionAuthEnv`) is the existing defense; this loader adds a defense-in-depth layer at the .env boundary. **No `.env` exists in the repo today** (loader must no-op when absent). **dotenv is NOT a dependency** (add `dotenv` for parse-only, or a minimal parser — Step-2.5).

## Acceptance criteria (what "done" means)
- [ ] A **pure, electron-free** loader module (compiles under `tsconfig.node.json` — L3; no `electron`/`app` import reaches the test) that parses a `.env` (given contents or a path) and returns the allowlisted vars to hydrate + a list of skipped keys (with reason).
- [ ] Called from `apps/desktop/main/index.ts` at startup **before** `startWorker()` reads `process.env`, loading the repo-root `.env` if present (**no-op when absent** — no throw).
- [ ] Hydrates `process.env[k]` **ONLY** for the recognized `SOW_*` allowlist: `SOW_MANAGE_TEMPORAL`, `SOW_TEMPORAL_ADDRESS`, `SOW_VAULT_ROOT`, `SOW_INGEST_WATCH`, `SOW_INGEST_WORKSPACE`, `SOW_WORKER_NODE`, `SOW_SUBSCRIPTION_ARM`, `SOW_SUBSCRIPTION_MODEL`, `SOW_EGRESS_ALLOWED_PROCESSORS`. Allowlist is a single named constant (documented to stay in sync with the `main`/`worker-host` reads).
- [ ] **⛔ SAFETY (load-bearing):** a shadowing key present in `.env` (any of the 13-var `SUBSCRIPTION_SHADOWING_ENV_KEYS` — `ANTHROPIC_API_KEY`, `HTTP_PROXY`, `all_proxy`, etc.) is **NEVER hydrated** into `process.env` (structurally — it's not on the `SOW_*` allowlist) **AND** surfaces a warning (skip + warn). Defense-in-depth atop `assertSubscriptionAuthEnv`.
- [ ] **Secrets (rule 7):** `VOYAGE_API_KEY`/`OPENROUTER_API_KEY` (or any non-`SOW_*` secret) present in `.env` are **NOT hydrated** (excluded by the allowlist) — Keychain remains the sole secret path. (Per the lead report: this is the confirmed posture, NOT a load-with-warning dev path.)
- [ ] Any non-allowlist key is skipped + warned (generic); a known shadowing key gets an escalated warning message.
- [ ] Does **NOT override** an already-set `process.env` value (a real shell/CLI export wins over `.env`) — default-safe.
- [ ] Warnings never print a secret/value — the warn names the KEY only (rule 7 / §16 redaction).
- [ ] `dev.sh` reconciled (the `source .env` block removed — becomes `exec pnpm --filter @sow/desktop dev`) + a `.env.example`/header note. **Flag at Step 9** (dev.sh is root-level/orchestrator territory — orchestrator edits it at round close; do NOT edit it in the slice unless told).
- [ ] `/preflight` clean.

## Wiring / entry point (Step 7.5)
Production entry: Electron main `startWorker()` / `app.whenReady()` init in `apps/desktop/main/index.ts` — the loader runs at module/startup init, BEFORE the `const config: WorkerHostConfig = {...}` block reads `process.env["SOW_*"]` (`:73-90`), so the hydrated allowlist vars flow through the existing "main reads env → threads over IPC → worker" path unchanged. No new IPC surface. Reached on every app launch; a no-op when `.env` is absent (today's default).

## RED test outline (Step 2)
1. **`loads_only_allowlisted_sow_keys`** — `.env` with `SOW_VAULT_ROOT=/x` + `RANDOM_KEY=y` ⇒ hydrate has `SOW_VAULT_ROOT`, NOT `RANDOM_KEY`.
   - Why: allowlist gate — no blanket load (the §5/§16 exclusion enforced by construction).
2. **`shadowing_key_in_env_is_never_hydrated`** (⛔ the load-bearing safety pin) — `.env` with `ANTHROPIC_API_KEY=sk-x` + `HTTP_PROXY=…` + `all_proxy=…` ⇒ NONE hydrated + each surfaces a shadowing warning; assert `process.env.ANTHROPIC_API_KEY` unchanged/absent post-load.
   - Why: §5 — a plaintext `.env` must not shadow the subscription / redirect egress (defense-in-depth over `assertSubscriptionAuthEnv`).
3. **`secrets_not_hydrated_from_env`** — `.env` with `VOYAGE_API_KEY`/`OPENROUTER_API_KEY` ⇒ NOT hydrated (Keychain-only).
   - Why: §16 rule 7 — secrets stay Keychain-resolved.
4. **`missing_env_is_noop`** — no `.env` / unreadable ⇒ no throw, empty hydrate, `process.env` untouched.
   - Why: default-safe (today's no-`.env` state).
5. **`does_not_override_existing_process_env`** — `SOW_VAULT_ROOT` already set in `process.env` ⇒ `.env`'s value does NOT override it.
   - Why: a real shell/CI export wins; no silent stale-`.env` override.
6. **`warning_names_key_not_value`** — a skipped shadowing/secret key's warning contains the KEY, never the value.
   - Why: §16 redaction — never log a secret.

## Things to flag at Step 2.5
1. **`dotenv.parse` vs a minimal parser.** My default vote: **`dotenv.parse(buf)`** (parse-only — it returns an object and NEVER auto-populates `process.env`; battle-tested for quotes/comments/multiline) + explicit allowlist-hydrate; add `dotenv` as a devDep. Alternative: a ~20-line minimal `KEY=VALUE` parser (no dep). Pick per the dep-vs-robustness trade; either MUST hydrate only the allowlist.
2. **Override policy.** My default vote: **do NOT override an already-set `process.env`** (existing shell/CI export wins; standard dotenv posture). Flag if the owner expects `.env`-wins.
3. **Shadowing-warn source.** My default vote: **generic skip-warn on ALL non-allowlist keys** (satisfies "skip + warn" structurally), **escalated** for a known shadowing key by importing the canonical `SUBSCRIPTION_SHADOWING_ENV_KEYS` from `@sow/worker` (single source of truth). ⚠ Confirm that value-import is clean (doesn't drag node-heavy worker deps into the main tier — LESSONS §5); if it does, fall back to generic-only warn + an in-code note referencing the worker set (safety is unaffected — the allowlist is the real gate).
4. **`.env` location.** My default vote: **repo-root `.env`** (matches `dev.sh`'s current source path). A packaged-app userData location is a future concern.
5. **Allowlist single-sourcing.** My default vote: a **named allowlist constant** in the loader + a comment to keep it in sync with the `main`/`worker-host` `SOW_*` reads. Extracting a fully shared `RECOGNIZED_SOW_ENV_KEYS` constant the reads also reference is a nice-to-have — flag if you want it in-scope.
