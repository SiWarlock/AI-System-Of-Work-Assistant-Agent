# /tdd brief — subscription_extraction_runtime_runner

## Feature
Bind the currently fail-closed **runtime branch** of the broker's real run leg (`provider-runner.ts:271-274`, the `!("provider" in route)` deny) to a subscription-backed extraction runtime runner that drives `createClaudeSubscriptionCompletion`, so an extraction route resolving to the Claude subscription produces a real `agent_extraction` candidate — all DORMANT (built + fake-tested, reachability-waivered; the shipped matrix route is unchanged, so nothing fires until the owner ENABLE).

## Acceptance criteria (what "done" means)
- [ ] A new worker-side factory (e.g. `createSubscriptionExtractionRunner(deps): ProviderRunner`) that, given a runtime route + an extraction job + an `EnforcedBudget`: resolves inline content via the injected content seam, builds the `CompletionRequest` via `buildExtractionCompletionRequest` (meeting-vs-source prompt selected on `job.capability`), calls `ClaudeSubscriptionCompletion.complete(req, signal)`, and on success returns `ok({ value: makeAgentResult({ status, candidateOutput: out.structuredOutput, usage, logs }) })`.
- [ ] The `structuredOutput` is passed through as `candidateOutput` UNVALIDATED (candidate data, rule 2) — the runner does NOT parse/normalize it; the broker schema gate + `bySchemaIdNormalizer` + `validateNoInference` own that. Pin: a completion whose `structuredOutput` is `{fields:{owner:{value:"TBD"}}}` yields an `AgentResult` whose `candidateOutput` deep-equals it.
- [ ] `CompletionError → GateDeny` mapping (redaction-safe, KIND-only message — never the SDK message, rule 7 / §16): a `budget` kind → a budget-flavored deny (`branch:"cancelled_budget"` or the existing budget deny shape); `auth`/`rate_limited`/`timeout`/`transport`/`malformed` → the appropriate retryable/terminal `provider_*` deny using `error.retryable`; `cancelled` → `provider_cancelled`. Reuse/extend the existing `denyFromProviderError`/`denyUnavailable` helpers where the shapes fit.
- [ ] The runner NEVER throws across the boundary (§16): a rogue throw (content seam, client) folds to a fail-closed `denyUnavailable` (redaction-safe, no cause echoed) — mirror the provider branch's outer `catch` (provider-runner.ts:304-308).
- [ ] **Budget:** the runner passes `budget.maxCostUsd` (the broker's `EnforcedBudget`) into `buildExtractionCompletionRequest`'s `opts.maxCostUsd` → the SDK `maxBudgetUsd` (the COST-1 dollar cap for the runtime route). Pin: an enforced `maxCostUsd` reaches the `CompletionRequest` the client receives (spy the injected client).
- [ ] **Selection / dormancy:** `createRealProviderRunner` routes a `!("provider" in route)` runtime route to the subscription runner ONLY when the subscription deps are bound; unbound ⇒ the EXISTING fail-closed `denyUnavailable` (byte-equivalent to today). The subscription deps ride the SAME default-OFF `ProviderTransportGate` arming as the provider branch — a factory-spy proves the subscription client/content seam is constructed ZERO times on the OFF/default path.
- [ ] Byte-equivalence: with the transport gate unset (shipped default), `assembleBackends`/`bootWorker` behavior is byte-identical to pre-slice (the runtime branch still denies; no subscription client constructed). Pin with a factory-spy zero-invocation test (L23/L27/L58 pattern).
- [ ] Reachability: the runner is WAIVERED (L11) — documented in-code + at Step 7.5 (the production `{runtime}` route + real content resolver bind at the owner ENABLE / #13). `/wired` on the runner traces to `createRealProviderRunner`'s runtime branch (constructed on the armed path), not to a live production dispatch.
- [ ] All unit tests in `apps/worker/test/composition/subscription-extraction-runner.test.ts` (+ any `provider-runner.test.ts` additions) pass.
- [ ] `/preflight` clean (repo-wide).
- [ ] security-reviewer + code-quality-reviewer run at Step 8 (this slice is `invariant` — see below).

## Things to flag at Step 2.5
1. **Content-resolution seam shape (the central design question).** The subscription needs inline `userPrompt` text; no `contextRefs → text` resolver exists. Options: (a) a NEW injected `ContentResolver` port `resolve(job/contextRefs): Promise<Result<string, …>>` (fake in tests; the real resolver — reading the transcript/`SourceEnvelope.body`/vault refs — binds at ENABLE); (b) thread the already-inline `SourceEnvelope.body` down to the run leg (works for source, not meeting; meeting has no inline body on the job today). My default vote: **(a) a new injected `ContentResolver` seam, left UNBOUND this round (fake-tested, real binding at ENABLE)** — it's uniform across meeting+source, keeps the run-leg signature intact, and matches the project's "build the seam dormant, bind the real one at the owner crossing" discipline (like the fetch transport). Flag (b) if you find `SourceEnvelope.body`/transcript already reaches the run leg by a path I missed. **This is a design choice, not a safety call — take the default unless you find a cleaner existing seam.**
2. **rule-5 egress veto for the cloud subscription route — confirm it's an ENABLE precondition, not this slice.** A subscription is CLOUD egress; today `source.process` routes `egressClass:"local"` (clears the veto via loopback). When the ENABLE changes the route to a cloud `{runtime}` route, the egress veto must fail closed for untrusted employer-raw source jobs with ack OFF (rule 5, no cloud fallback). Because this slice does NOT change the route (stays local/ollama), there is nothing to arm here — but **verify the veto STILL precedes the run leg for a runtime route** (it runs on the job/route before the run leg regardless of branch) and add a Step-9 arming-precondition note ("#13: the cloud runtime route re-triggers the egress veto — prove fail-closed for employer-raw source before ENABLE"). My default vote: **treat the veto re-eval as a #13 arming precondition (note it), do NOT change the route or the veto in this slice.** Escalate to the orchestrator ONLY if you find the veto does NOT run for a runtime-branch route (that would be a live safety Finding).
3. **Where the subscription deps bind — extend `RealProviderRunnerDeps`, or a sibling gate?** Options: (a) add optional `subscription?: { completion: () => ClaudeSubscriptionCompletion; content: () => ContentResolver; model; betas? }` to `RealProviderRunnerDeps` (unbound ⇒ runtime branch keeps denying); (b) a separate arming sub-gate. My default vote: **(a)** — one arming bundle (`buildRealProviderTransportGate`) already assembles the crossing deps (L58); add the subscription deps there, factory-thunked (zero-invocation on OFF). Keep the composition-root edit minimal + its own commit if byte-equivalence is at risk.
4. **Meeting-vs-source discrimination key.** Default vote: **`job.capability`** (`meeting.close` vs `source.process`) selects `buildMeetingExtractionRequest` vs `buildSourceExtractionRequest`. Confirm the capability values are stable/available on the `AgentJob` at the run leg (they are — `job.capability`).
