# /tdd brief — workspace_onboarding_provisioning (14.1 worker leg)

## Feature
A **real production workspace-provisioning path** (`provisionWorkspace`) that mints a workspace — persisting the validated `Workspace` aggregate into the existing durable `WorkspaceConfigRepository` AND unioning its id into the fail-closed WS-8 `{workspaceIds}` registry — plus the `onboarding` tRPC procedure that drives it. This is the production replacement for the dev-only `provisionDevWorkspace` fixture: a user can now EXIST in the system without a `devProvision` boot spec. Worker/db-track only; the desktop onboarding UI is a later desktop-impl slice.

## Acceptance criteria (what "done" means)
- [ ] `provisionWorkspace(deps, spec)` builds a **validated** `Workspace` via `defaultWorkspace(...)` (safe-default posture: egress CLOSED, `isolated` visibility — safety rule 5), `upsert`s it into `WorkspaceConfigRepository`, and unions its id into the fail-closed `{workspaceIds}` registry — with **NO** dev checkbox-parse / read-model card seeding (that stays `provisionDev`-only).
- [ ] `spec` carries the onboarding inputs: workspace identity (`id`, `name`, `type`), the chosen **vault root** (→ the existing `Workspace.markdownRepoPath` — no frozen-seam change), `gbrainBrainId`, and the chosen **preset** (see Step-2.5 Q1 for where preset lands).
- [ ] **WS-8 sole-authority pin (lead ruling #1):** a workspace with a persisted `WorkspaceConfig` row but **NOT** unioned into the registry ⇒ `resolveKnownWorkspace` returns `false` ⇒ still **zero read path**. The config store is never a visibility source.
- [ ] **After provisioning, before/after asymmetry:** a workspace-scoped read for the new id fails closed BEFORE `provisionWorkspace`, resolves AFTER (it is now a registry member).
- [ ] **Registry union discipline (mirror `registerWorkspace`):** union preserves prior members (no drop); idempotent (re-provisioning an existing id is a no-op union + an overwrite upsert); a genuine store fault ⇒ typed err, **never** fold-to-empty (would drop known workspaces → fail their scoped reads closed).
- [ ] **provisionDev still works (lead ruling #2):** `provisionDevWorkspace`'s existing tests stay green; if the registry-union helper is extracted (Step-2.5 Q4), `provisionDev`'s behavior is byte-identical — pin it.
- [ ] The `onboarding` tRPC procedure (`apps/worker/src/api/procedures/onboarding.ts`) is **registered in the worker router**, loopback + session-token authed like the sibling command procedures, and calls `provisionWorkspace`; a create round-trips (returns a typed provisioned-workspace summary — id + registry-member=true), a second create for the same id is idempotent.
- [ ] **Redaction (§16 / safety rule 7):** the procedure returns typed `Result`s only; never echoes secrets/raw content; a fault crosses as a stable code, never a driver cause.
- [ ] **Reachable from** the router-registered `onboarding` procedure → `provisionWorkspace` → `backends.repos.workspaceConfig` + registry (`/wired`-style, from a production entry point — not test-only). The desktop onboarding UI caller is a NAMED later slice.
- [ ] `/preflight` clean (worker + repo-wide turbo typecheck/test).

## RED test outline (Step 2)
In `apps/worker/test/composition/provisionWorkspace.test.ts`:
1. **`provision_upserts_workspace_and_registers`** — Asserts: `provisionWorkspace` calls `workspaceConfig.upsert` with a validated `Workspace` (egress CLOSED via `defaultWorkspace`) AND unions the id into the registry read-model; returns ok. Why: §19.1 real provisioning path; §5 registry membership.
2. **`config_row_without_registry_is_zero_read_path`** — Asserts: seed a `WorkspaceConfig` row but do NOT union → `resolveKnownWorkspace` = false. Why: **lead ruling #1** — registry is the sole visibility authority (safety rule 4).
3. **`scoped_read_fails_closed_before_resolves_after`** — Asserts: before `provisionWorkspace`, a scoped read fails closed; after, it resolves. Why: §5 WS-8 fail-closed → known transition.
4. **`registry_union_preserves_prior_members`** — Asserts: provisioning B after A leaves A in the set. Why: no-drop union (mirror `registerWorkspace`).
5. **`registry_union_idempotent`** — Asserts: re-provisioning A is a no-op union + upsert overwrite (no duplicate id). Why: idempotent onboarding.
6. **`registry_get_fault_is_typed_err_not_empty`** — Asserts: a store fault on the registry get ⇒ typed err, NOT a fold-to-empty. Why: fault ≠ benign miss (would drop known workspaces).
7. **`no_read_model_card_seeding`** — Asserts: `provisionWorkspace` writes NO `workspace_cards`/`project_cards`/`recent_changes` rows (that is `provisionDev`'s dev-unlock job). Why: production provisioning ≠ dev seeding.
8. **`default_egress_closed`** — Asserts: the upserted `Workspace` has `employerRawEgressAcknowledged=false` + empty processors + `isolated` visibility. Why: safety rule 5 fail-closed default.

In `apps/worker/test/api/procedures/onboarding.test.ts`:
9. **`onboarding_create_round_trips`** — Asserts: the procedure calls `provisionWorkspace` and returns a typed provisioned summary (id + member=true). Why: §19.1 onboarding procedure; §11 first-run.
10. **`onboarding_create_idempotent`** — Asserts: a second create for the same id succeeds idempotently. Why: re-entrant onboarding.
11. **`onboarding_error_is_typed_no_raw`** — Asserts: a provisioning fault surfaces a stable code, no driver cause / raw content. Why: §16 / safety rule 7.

Plus a `provisionDev` regression pin (Q4): the existing `provisionDev` tests stay green; if the union helper is extracted, add one assertion that `provisionDevWorkspace` still unions identically.
