## Acceptance criteria (what "done" means)
- [ ] A `node:fs` watcher on a configured vault root, given a `.md` file ADD or CHANGE, captures it via `createFileReadTransport(root)`→`extractFileSource` and dispatches a `sourceIngestion` run via `dispatchSourceIngestion` with `trigger:"connector_event"`.
- [ ] **WS-2 binding (no inference):** the captured source's `workspaceId`/`sensitivity` come from the watcher's configured policy binding for that vault root (a vault-per-workspace mapping), NOT from the file path or content; `sourceId` is a deterministic function of the path. (REQ-F-002 / REQ-F-017.)
- [ ] **ROOT-confinement (reuse C2):** an event whose resolved real path escapes the vault root (symlink-escape) is IGNORED — never captured; the transport read is already root-confined, so containment is double-guarded.
- [ ] **Debounce:** rapid repeated events on the same path (editor multi-write) coalesce to ONE dispatch per debounce window; idempotent dispatch (C3a workflowId dedupe) backstops any residual double-fire.
- [ ] **Scope:** `.md` add/change only; a delete/rename-away is IGNORED (deletion→retention is a separate flow, not C3b).
- [ ] **Degraded-safe:** when Temporal is unavailable, the watcher still runs but the dispatch fails-closed (C3a's typed err + `worker_down` §16 item) — no crash, no silent-drop, no unbounded growth; the watcher NEVER throws out of its event handler.
- [ ] **Boot composition:** `bootWorker` builds a loopback Temporal Client when Temporal is available + starts the watcher; degraded-safe when not; a clean shutdown stops the watcher.
- [ ] **`SOW_TEMPORAL`-gated** integration test: a temp vault + the REAL watcher → write a `.md` file → a `sourceIngestion` run dispatches + reaches the deterministic terminal state (the full end-to-end make-it-real proof). Fast-unit tests cover filter/debounce/containment/degraded with no server.
- [ ] Default suite unaffected; repo-wide `turbo typecheck test` green; `/preflight` clean.

## Things to flag at Step 2.5
1. **Watcher mechanism.** Default vote: **`node:fs.watch(root, {recursive:true})`** (native, no new dep; macOS supports recursive) + per-path debounce; re-stat to classify add/change vs delete. Flag the known `fs.watch` caveats (duplicate/again events → the debounce + the C3a idempotency dedupe both absorb them). Alt (chokidar) — avoid a new dep unless fs.watch proves unreliable.
2. **WS-2 workspace binding.** Default vote: **the watcher is constructed with an explicit `{ root, workspaceId, sensitivity }` policy binding** (a vault-per-workspace config) — the captured source takes those verbatim; `sourceId` = a deterministic hash/derivation of the path. NEVER infer workspace from content/path (REQ-F-017). This is the safety crux.
3. **Degraded lifecycle.** Default vote: **the watcher runs independent of Temporal; a dispatch when Temporal is down fails-closed via C3a (typed err + health item), NOT a durable queue** (a persistent retry/outbox is out of C3b scope — note it as a follow-up). The watcher never crashes on a down Temporal.
4. **Debounce window + event classification.** Default vote: **a short per-path debounce (~150–300ms)**; treat `change` + `rename`-with-file-present as add/change, `rename`-with-file-absent as delete (ignored). Confirm the exact `fs.watch` event semantics on macOS.
5. **Boot Client vs test client.** Default vote: **`boot.ts` builds a loopback `@temporalio/client` Client (available when Temporal up); the gated test injects the TestWorkflowEnvironment client** into the same watcher-start seam — so the watcher+dispatch path is genuinely tested, not dormant.
