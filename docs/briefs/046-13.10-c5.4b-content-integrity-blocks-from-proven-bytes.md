# /tdd brief — c5.4b_content_integrity_blocks_from_proven_bytes

## Feature
Close **C5.4b GO-LIVE PRECONDITION 1 (content integrity)**: extend the internal `CopilotServingVerdict` gated arm to carry each admitted citation's **rehydrated proven bytes** (`AdmittedFact.content` + `mdContentSha`), and have the provenance-stamping decorator **REBUILD `RetrievedContext.blocks` from those proven bytes** for the admitted set. Today the decorator stamps `sources[].provenance = knowledge_writer` but the model synthesizes over `blocks[]` — a **separate array** the inner adapter populates with unverified bytes. So a `trusted` label can sit over unverified block content. This slice makes the two agree: when the serving gate proves a source, the bytes the model reads for it are the proven ones.

**SAFETY-CRITICAL** (the Copilot content-trust gate — the whole propose path's trust root). Own commit, no bundling. **Propose stays OFF the entire time**: boot wires only `createInterimDegradedServingOracle`, which always returns `degraded_direct_markdown` ⇒ the new rebuild path is never taken on the live path today (exercised only by fake gated verdicts in tests + the future real oracle). No flag flip, no hard-line crossing.

## Things to flag at Step 2.5
1. **Partial-admission block policy.** When a context is partially admitted (some sources proven, some not — an inherently read_only job since content-trust needs ALL sources `knowledge_writer`): **(A)** rebuild `blocks` to the proven subset only, drop the unverified inner blocks; **(B)** keep unverified inner blocks for the unadmitted sources alongside the proven ones. My default vote: **A** — the invariant "no `knowledge_writer` stamp coexists with unverified blocks the model reads" is simpler to reason about and strictly safer; the read-quality loss is only for untrusted context on a read-only ask, and today (interim oracle) this path is never taken live anyway.
2. **Admitted-map value shape.** `ReadonlyMap<citationId, { content, mdContentSha }>` vs a richer per-citation record vs `content`-only. My default vote: **`{ content, mdContentSha }`** — matches the precondition-1 text verbatim and keeps `mdContentSha` available for future belt-and-suspenders, at ~zero cost.
3. **Does the decorator re-hash `content` against `mdContentSha`?** The gate already proved `content`'s hash == the canonical page sha (content-hash-match leg) + HMAC signature before admitting, so re-hashing in the decorator duplicates the gate and pulls a hashing dep into the worker. My default vote: **carry-only, do NOT re-hash** — the gate is authoritative; `mdContentSha` rides along for traceability, not re-verification. (Reviewer: confirm this doesn't weaken the trust root.)
4. **Rebuilt-block ordering + multi-fact citations.** Order the rebuilt `blocks` by **retrieved source order** (stable, deterministic) filtered to admitted. For a citation resolving to >1 factId (NOT the production page-only resolver, but the core is generic): default = the admitted page fact's content; if the resolver ever returns multiple content-bearing facts, concatenate their `content` in resolved order and use the first/page fact's `mdContentSha`. My default vote: **retrieved-source order; page-fact content in the production one-fact case.** Flag if the fake-oracle tests force a decision here.
