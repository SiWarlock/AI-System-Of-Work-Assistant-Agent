Adopt pgvector for the local knowledge graph

Decision #architecture #knowledge #database KnowledgeWriter · rev 0b9 · 2026-06-30 · Employer-Work/architecture/ Canonical Markdown — edit in Obsidian
Status: Accepted · 2026-06-30
Decision

Use pgvector (the Postgres extension) as the vector store for the local knowledge graph, instead of running qdrant.

Context

The GBrain needs a local vector store for embeddings. Running a second database service adds operational + backup burden on a single-user, local-first machine.

Rationale
  • Keeps everything in the local Postgres we already run — one service to operate, migrate, and back up.
  • pgvector recall + p95 latency are competitive at our scale (~1M vectors).
  • Simpler failure + recovery story — one store, one backup.
Alternatives considered
  • qdrant — rejected: a second service to run and back up.
  • sqlite-vss — rejected: maturity + concurrency concerns.
Consequences
  • GBrain writes embeddings into Postgres via pgvector.
  • Add the pgvector extension to the DB migration.
